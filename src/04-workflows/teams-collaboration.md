# Teams 多进程协作

> 超越单进程限制：多个独立 Agent 进程通过消息传递机制协同工作，构建大规模分布式 AI 编程流水线。

## 文章概述

当单进程无法满足复杂工程场景的需求时，Teams 多进程协作提供了解决方案。与 Agent 派生（父子关系）不同，Teams 是多个独立 Agent 进程的平级协作——每个 Team 成员都是独立的 Agent 实例，有自己的上下文、技能和生命周期，通过消息传递机制通信和同步。

本文从 Teams 的架构设计原则入手，解释为什么需要从单进程走向多进程协作。你会理解 Team 的成员角色定义、通信协议（`team_send_message` 的工作原理）和消息类型体系（任务分配、状态同步、结果汇总）。我们深入讨论进程内集群的优缺点——多个 Agent 在同一个进程内协作时，资源竞争和隔离问题如何管理，集群的生命周期如何维护。然后通过对比表分析 Team Mode 与独立 Agent 的适用场景，帮助你在性能与隔离性之间做出权衡。最后，讨论大规模 Teams 的工程实践：分层架构设计、监控和日志收集、故障恢复策略。

## 内容要点

1. **Teams 架构概述** — 从单进程到多进程：为什么需要 Teams 架构；Team 的成员角色定义和职责划分；通信协议的基础设计（消息类型、队列、优先级）。
2. **消息传递机制** — `team_send_message` 的工作原理和底层实现；消息类型：任务分配（Task Assignment）、状态同步（Status Sync）、结果汇总（Result Aggregation）；消息队列的优先级管理和超时机制。
3. **进程内集群** — 多个 Agent 在同一进程内协作时的资源竞争和隔离策略；集群的生命周期管理（创建、扩容、缩容、销毁）；共享状态的一致性问题（最终一致性 vs 强一致性）。
4. **Team Mode vs 独立 Agent** — 适用场景对比表：性能、隔离性、资源消耗、复杂度、故障容错五个维度的权衡分析；混合模式设计思路——部分 Team 成员在进程内运行，部分独立部署。
5. **大规模 Teams 的工程实践** — 分层 Team 架构（父 Team 包含子 Team 的分层治理模型）；Team 监控和日志（消息轨迹追踪、Agent 心跳检测、性能指标采集）；故障恢复策略（成员宕机后的任务重新调度、消息重试、降级方案）。
6. **Teams 的设计哲学** — Teams 是 Agent 协作的操作系统——提供通信、调度、状态管理的完整基础设施；内部通信使用消息传递，对外输出使用文件（WORKFLOW_STATE.md）。
