# Agent 派生模式

> 父 Agent 动态生成子 Agent 处理子任务——子Agent、委派、协调者三种派生模式的设计原理和工程实践。

## 文章概述

Agent 派生是扩展单一 Agent 能力边界的关键机制。当一个 Agent 面对超出自身能力范围的任务时，它可以派生子 Agent 来分担工作。派生不是替代，而是能力延伸——子 Agent 继承父 Agent 的上下文和权限，专注于完成特定的子任务。

本文介绍三种 Agent 派生模式：子 Agent 模式（父 Agent 创建子 Agent 执行独立的子任务）、委派模式（将特定领域任务委托给专门化的 Agent 处理）和协调者模式（一个协调者 Agent 分配和汇总多个 Agent 的输出）。我们深入分析 `task()` API 的派生实现机制——`category` 参数如何选择派生目标、`load_skills` 如何传递技能上下文、结果如何合并。此外，还会对比《马书》中的 Agent 派生框架与 OpenCode 的实现差异，帮助你理解不同设计哲学的优劣。最后讨论派生模式的工程实践要点：深度限制（避免递归失控）、权限隔离（子 Agent 不能越权）和错误传播策略。

## 内容要点

1. **派生模式的概念** — 为什么需要派生：单一 Agent 的能力边界和认知负载限制；派生与协作的区别和配合方式——派生是纵向扩展（父子关系），协作是横向扩展（平级关系）。
2. **三种派生模式** — 子 Agent 模式：父 Agent 动态创建子 Agent 执行子任务，子任务完成即销毁；委派模式：将专业领域任务委托给专门训练过的 Agent 处理；协调者模式：一个协调者 Agent 负责任务分配、进度跟踪和结果汇总。
3. **task() API 的派生实现** — `category` 参数选择派生目标 Agent 类型；`load_skills` 向子 Agent 传递技能上下文；返回值处理和结果合并策略（加权平均、投票表决、顺序拼接）。
4. **马书 Agent 派生框架对比** — 《马书》三种派生模式与 OpenCode 实现的映射关系；差异分析（上下文继承深度、权限颗粒度、错误处理策略）；可借鉴的设计思想。
5. **派生模式的工程实践** — 派生深度限制的配置方法和默认值；派生 Agent 的权限隔离原则（最小权限、职责分离）；错误传播机制（子 Agent 失败如何影响父 Agent 的执行路径）；派生 Agent 的生命周期管理和资源回收。
