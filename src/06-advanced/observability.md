# 可观测性

> 不知道 Agent 在做什么、花了多少 Token、为什么出错，就无法调优和运维。日志、指标、追踪三支柱构建完整的可观测性体系。

## 文章概述

当 AI 编程流水线从个人工具升级为团队基础设施时，可观测性就变成了刚需。你需要知道：每个 Agent 任务花了多长时间、消耗了多少 Token、调用了哪些工具、有没有报错、性能有没有退化。没有这些数据，调优就是盲目的，出问题就是靠碰运气排查。

本文从可观测性的三大支柱出发——**日志**（记录了什么事）、**指标**（发生了多少次）、**追踪**（完整链路是什么样的）。然后介绍 5 层遥测架构（Agent 层 / Session 层 / 工具层 / 网络层 / 系统层），每层的关注点和关键指标。接着深入 `logEvent` 系统——事件格式和结构、过滤和聚合、输出方式（控制台 / 文件 / 外部系统）。在生产级监控方面，讨论关键指标面板设计、告警规则配置（Token 消耗异常 / 错误率上升 / 响应时间超标）、性能基准和趋势分析。最后展示如何基于可观测性数据做性能优化——从日志发现瓶颈、从指标优化成本、从追踪定位错误。

## 内容要点

1. **可观测性的 3 个支柱** — 日志（Logging）：结构化的时间序列事件，记录 Agent 执行的每个步骤。指标（Metrics）：可聚合的量化数据（调用次数、Token 消耗、响应时间、错误率）。追踪（Tracing）：从用户请求到 Agent 执行的完整链路，跨 Session 关联。三支柱的关系和协同方式。

2. **5 层遥测架构** — Agent 层（模型调用、Token 消耗、Agent 切换）、Session 层（会话持续时间、轮次数量、任务类型分布）、工具层（工具调用频率、调用结果、错误率）、网络层（API 延迟、重试次数、带宽消耗）、系统层（CPU/内存使用、进程状态）。每层的关键指标和可观测性关注点。

3. **logEvent 系统** — 事件的格式和结构（时间戳、层级、类型、Payload），事件的过滤和聚合（按层级/类型/时间范围筛选），事件输出方式（控制台实时输出 / 文件归档 / 外部系统转发如 ELK/Datadog）。

4. **生产级监控** — 关键指标面板布局（Token 消耗趋势图、错误率仪表板、响应时间分布），告警规则配置（Token 消耗异常告警、错误率突增告警、响应时间超标告警），性能基准和趋势分析（基于历史数据建立基线，自动检测异常偏离）。

5. **基于可观测性的优化** — 从日志发现性能瓶颈（哪些步骤耗时最长），从指标优化成本（Token 消耗在哪里最高），从追踪定位错误（错误发生时的完整上下文和根因分析）。
