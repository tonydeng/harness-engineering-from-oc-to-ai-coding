# 案例二：遗留系统现代化

> 对一个拥有 247 个文件、186 个依赖、34K 行代码的遗留 Node.js 项目进行全面安全审计和渐进式改造。展示"改造而非重写"的工程智慧。

## 案例概述

现实世界中，我们面对更多的是遗留系统而不是绿地项目。本案例选取了一个典型的遗留 Node.js 项目：代码规模 34K 行，依赖数量 186 个，没有测试覆盖，存在大量已知的安全漏洞，多个核心依赖已经停止维护。团队面临的挑战不是"要不要改"，而是"怎么改"——既要修复安全问题，又要保持业务不中断。

案例采用"审计 → 计划 → 执行 → 验证"的闭环策略，这与传统的"评估 → 重写"思路有本质区别。第一阶段用 `/init-deep` 进行深度初始化和全量扫描，生成项目现状报告。第二阶段加载 5 人 `security-research` 团队，从静态分析、依赖审计、认证审查、基础设施审计和合规检查 5 个维度并行审计，输出带 CVSS 评分的审计报告。第三阶段基于审计报告制定分阶段修复路线图，按"紧急修复 → 重构加测试 → 依赖升级 → 体系建立"四期执行。

这个案例的核心价值在于证明了 Harness Engineering 方法对遗留系统同样有效——甚至更有价值。增量改造能在不中断业务的前提下系统性降低技术债，而安全审计的自动化让过去需要几周的人工审查缩短到几小时。

## 内容要点

1. **项目现状** — 遗留项目的详细画像：文件构成、依赖树分析、已知安全漏洞清单。让读者理解"遗留系统"的真实面目以及为什么不能简单重写。

2. **阶段一：全景扫描** — 使用 `/init-deep` 进行深度初始化，通过 `@explore` 执行全量扫描（文件统计、依赖分析、安全扫描）。生成结构化的项目现状报告，这是后续所有决策的数据基础。

3. **阶段二：安全审计** — 加载 `security-research` 团队，5 人并行执行 5 维审计。展示经典安全发现的代码示例（SQL 注入、XSS、JWT 未验证签名）以及 CVSS 评分的自动化生成方法。审计阶段也包含 OpenCode 自身安全配置检查和安全基线验证。

4. **阶段三：重构计划** — 基于审计报告制定分阶段修复路线图。四期规划各有明确目标和交付标准，按 CVSS 评分加业务影响排序。这一阶段包含技术债量化评估和架构风险评估。

5. **阶段四：增量改造** — 使用 7-Agent Pipeline 执行安全修复，添加单元测试和集成测试，执行依赖安全升级。核心原则是"每次改动都可验证、可回滚"，包含灰度策略的实现说明。

6. **阶段五：验证与交付** — 全面回归测试、二次安全审计、改造前后量化对比。用数据证明改造的效果，包括漏洞数量变化、测试覆盖率提升、依赖安全评分改善。
