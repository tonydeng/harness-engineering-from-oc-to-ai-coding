## 二、T1：内部 Markdown 链接标准化

### 2.1 需求背景

用户要求将所有内部 Markdown 链接标准化为符合 Docsify（`relativePath: false`）解析的正确相对路径格式。

**关键约束**：
- Docsify v4 默认以当前页面的 **URL 路径**（而非文件系统）为基准解析相对链接
- 根级文件（`src/README.md`、`_sidebar.md`）的链接规则与子目录文件不同
- 不修改外部 URL 或锚点链接
- 验证每个目标文件存在

### 2.2 意图识别与编排决策

```
Context: 用户要求审计全部内部 Markdown 链接并标准化
Intent: Investigation + Fix — 需要先全面审计，再批量转换
Routing decision: 初始审计用直接工具(grep/ast-grep) → 批量转换用 8 个并行 explore agent

第一阶段 — 发现关键规则：
  Docsify 在 relativePath: false（默认）模式下，
  从 /#/XX-YY/file 解析 (file.md) → /#/XX-YY/file ✓
  解析 (XX-YY/file.md) → /#/XX-YY/XX-YY/file ❌ 404
  解析 (../ZZ-WW/file.md) → /#/ZZ-WW/file ✓
```

### 2.3 执行过程

#### 第一步：初始审计

| 检查项 | 结果 |
|--------|------|
| 全部 .md 文件数 | 56 |
| 内部链接总数 | 262 |
| 断链数 | 0（全部有效） |
| 侧边栏链接 | 全部可解析，文件有对应条目 |
| 代码块格式问题 | 12 处修复（\`\`\`\`language:relative/path {行号}） |

#### 第二步：8 个并行转换 Agent（关键教训）

| Agent ID | 章节 | 目标 | 结果 |
|----------|------|------|------|
| (并行 ×8) | Ch00-Ch07 | 标准化链接路径 | ❌ **全部错误** |

**错误模式**：
- 同目录 `(file.md)` → `(chapter/file.md)` — 解析为 `/#/chapter/chapter/file`（404）
- 跨目录 `(../dir/file.md)` → `(dir/file.md)` — 解析为 `/#/source/dir/file`（404）

**根因**：子 Agent 假设的是文件系统路径模型，而不是 Docsify 的 URL 路径模型。所有 8 个 Agent 用了相同的错误假设。

**影响**：54 个文件、~150 处链接全部损坏。

#### 第三步：回滚修复

```python
# 修复策略 — Python 脚本批量修正
修复规则:
  1. (chapter/file.md)  → (file.md)           # ~120 处 — 同目录恢复裸文件名
  2. (other/file.md)     → (../other/file.md)  # ~30 处  — 跨目录恢复 ../ 前缀
  3. (README.md)         → (../README.md)      # 2 处   — 根 README 引用
  4. (chapter/README.md) → (README.md)         # 2 处   — 同目录 README 引用
```

#### 第四步：AGENTS.md 规范更新

```diff
- **内部链接**：相对路径（如 [text](path.md)）
+ - **内部链接（Docsify 规则）**：链接基于当前页 URL 路径解析（默认 relativePath: false）：
+   - **同目录**：[text](file.md) ✓
+   - **跨目录**：[text](../target-chapter/file.md) ✓ — 必须带 ../，否则 404
+   - **根 README**：[text](../README.md) ✓
+   - **根级文件**（src/README.md、_sidebar.md）：[text](chapter/file.md) ✓
+   - ❌ 严禁从子目录写 `](chapter/file.md)` 或 `](子目录/target.md)`
```

### 2.4 验证结果

- ✅ 262 条内部链接全部有效
- ✅ 侧边栏链接全部可解析
- ✅ 零处 `(chapter/file.md)` 残留（从子目录文件）
- ✅ `mdbook build`：0 errors
- ✅ AGENTS.md 规范更新完成

### 2.5 关键结论

- **框架级假设必须先验证**：子 Agent 对 Docsify 链接解析模型的理解错误导致了大规模损坏
- **并行度不总是好事**：8 个 Agent 基于相同的错误假设做转换，放大了错误的影响面
- **回滚计划应预先准备**：因为没有版本控制变更，修复脚本必须自行编写

## 三、T2：GitHub Actions Pages 部署失败修复

### 3.1 需求背景

用户发现 GitHub Actions 执行报错：
```
Error: Get Pages site failed. Please verify that the repository has Pages enabled and
configured to build using GitHub Actions, or consider exploring the `enablement` parameter
for this action. Error: Not Found - https://docs.github.com/rest/pages/pages#get-a-apiname-pages-site
```

### 3.2 意图识别与编排决策

```
Context: "组织团队成员对github action的执行错误进行修复" + 错误日志
Intent: Investigation + Fix — 先诊断根因，再修复
Routing decision: 诊断用 explore(API) + librarian(文档) 并行 → 直接修复 workflow

Agile-coach 工作流路径:
  头脑风暴 → 计划 → 实施 → 评审 → 验证
  风险等级: 中（单模块 CI/CD 配置修复）
```

### 3.3 诊断过程

**并行调查**：

| 调查项 | Agent/工具 | 结果 |
|--------|-----------|------|
| GitHub Pages API | `curl https://api.github.com/.../pages` | **404 Not Found** — Pages 未启用 |
| configure-pages action docs | `webfetch action.yml` | `enablement: true` 可用但需要 PAT（非 GITHUB_TOKEN） |
| Pages 部署流程 | `read deploy-docsify.yml` | workflow 正确，3 步骤（configure → upload → deploy） |
| librarian 深度研究 | `task(background, librarian)` | 确认 enablement 逻辑、token 要求、v6 版本 |

**根因确认**：仓库 `tonydeng/harness-engineering-from-oc-to-ai-coding` **未启用 GitHub Pages**。

**`enablement` 参数约束**：
- **默认 `false`**（v3+ 因安全 bug bounty 从 `true` 改为 `false`）
- 设为 `true` 时，调用 `POST /repos/{owner}/{repo}/pages` 创建 Pages site
- **但需要 PAT**（`GITHUB_TOKEN` 缺乏 `administration:write` 权限）

### 3.4 修复方案

#### 工作流修改

```diff
       - name: Setup Pages
         uses: actions/configure-pages@v5
+        with:
+          enablement: true
```

YAML 语法验证：✅ 通过

#### 用户操作路径（二选一）

| 路径 | 操作 | 复杂度 |
|------|------|--------|
| **A: 手动启用（推荐）** | Settings → Pages → Source: GitHub Actions | 30 秒，一次 |
| **B: 全自动化** | 创建 PAT → 存为 `PAGES_PAT` secret → 显式传入 token | 需 token 管理 |

### 3.5 验证

- ✅ YAML 语法 `python3 yaml.safe_load` 校验通过
- ✅ workflow 结构完整（checkout → configure → upload → deploy）
- ✅ 前置条件 `.nojekyll` 文件已存在
- ❌ 无法自动验证 GitHub Pages 是否已启用（需用户手动操作）

### 5.7 思考过程与结论记录

#### 关键决策点 1：链接修复方案

```
Options:
  A. 为每个文件计算正确的相对路径（从文件系统到目标）
  B. 统一规则：同目录裸文件名，跨目录 ../ 前缀

选择 B：
  理由: 规则简单、可验证、不依赖文件路径计算
  代价: 根 README 引用需要特例处理（从子目录必须 ../README.md）
  验证: grep 检查无 (chapter/file.md) 模式残留
```

#### 关键决策点 2：CI/CD 修复方案

```
Options:
  A. 添加 enablement: true（仅修改 workflow 文件）
  B. 添加 enablement: true + 通知用户手动步骤

选择 B：
  理由: enablement: true 需要 PAT，仅 GITHUB_TOKEN 不够
  通知内容: "请去 Settings → Pages → Source: GitHub Actions 手动启用"
  验证: YAML 语法校验通过即可
```

#### 关键决策点 3：回滚策略（T1 修复）

```
Options:
  A. 逐个文件手动 revert
  B. Python 脚本批量修正

选择 B：
  理由: ~150 处错误分布在 54 个文件中，手动修复不现实
  策略: 编写 Python 脚本用三种规则模式匹配替换
  验证: grep 确认零残留错误
```

### 5.8 关键教训

| # | 教训 | 场景 | 建议 |
|---|------|------|------|
| 1 | **框架级假设必须先验证** | 8 个 Agent 全基于错误假设工作 | 批量修改前先做 1 个文件的试运行验证 |
| 2 | **并行放大风险** | 同一错误假设被 8 个 Agent 同时放大 | 高影响任务先用 "dry-run" 模式做单个试运行 |
| 3 | **子 Agent 不理解隐式约束** | Agent 不知道 Docsify 的 URL 解析模型 | 在 prompt 中显式写明框架的链接解析规则 |
| 4 | **enablement 参数 ≠ 自动修复** | 需要 PAT 才能工作，仅加参数不够 | CI/CD 修复需同时告知用户剩余手动步骤 |
| 5 | **写作类任务不需要子 Agent** | 日志文档直接写比委托更快 | 元数据密集型写作优先用 session 工具收集数据后直接写 |
