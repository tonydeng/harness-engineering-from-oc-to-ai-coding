## 9. 经验教训与改进建议

### 做得好的

| # | 经验 | 说明 |
|---|------|------|
| 1 | **审计先于修复** | Sprint 2 审计发现 P0-1/P0-3/P0-4/P1-1 已被前序 Sprint 修复，避免重复劳动 |
| 2 | **并行度最大化** | Round 1 7 agent 并行评审（墙钟 ~2 分钟）、5 deep-research 并行验证、3 个最终方向并行 |
| 3 | **原子化 agent** | 每个 agent 聚焦 1 个视角/1 个章节，prompt 打包完整上下文（文章内容 + 审阅维度 + 输出格式） |
| 4 | **验证闭环** | 每子阶段末尾执行 mdbook build，确保零错误递进 |
| 5 | **agent 自我纠错** | 代码块格式 agent 在处理 workflow-patterns.md 时 sed 替换过广，被及时发现并修复 |
| 6 | **模块化审阅团队** | 7 个独立视角→综合共识→定向修复的流水线，避免单一审阅者的盲区 |

### 可改进的

| # | 不足 | 建议 |
|---|------|------|
| 1 | 部分人物视角 skill 在子 agent 不可用 | 子 agent 环境缺少部分用户自定义 skill，需提前 fallback 方案 |
| 2 | 代码块格式化 agent 运行 11+ 分钟 | 474 块单 agent 串行效率低，应拆分为按章节并行的多个 agent |
| 3 | 虚构模型名称发现晚 | 早期审阅就应纳入检查清单，而非等到全书审阅才发现 |
| 4 | 跨 session 审计信息重复 | Sprint 2 审计结果存储在会话压缩摘要中而非持久文件，后续 session 需重新 grep |
