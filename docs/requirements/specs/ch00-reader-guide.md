# Ch0: 读者导航

## 概述

第 0 章是全书唯一的"元内容"章节——不讲授任何 Harness Engineering 知识，而是帮助读者回答三个核心问题：

1. **"这本书适合我吗？"** — 通过角色自诊和前置知识检查，快速判断是否值得投入时间
2. **"我该读哪些部分？"** — 基于 13 种读者角色的定制化路径，跳过不相关的内容
3. **"怎么读效果最好？"** — 两种阅读模式的选择和实操建议

本章不假设任何技术背景，是所有读者的第一站。即使已经熟悉 OpenCode 的用户，也建议花 5 分钟浏览 Ch0，了解全书的整体架构和版本信息，避免后续遇到版本不匹配的问题。

**章节核心主题**：帮助每一位读者在 5 分钟内判断"值不值得读"+"怎么读最省时间"——这是全书的后设导航层。

> **章节规模**：3 篇文章（1 现有 + 2 基本完成），本章重点在于 **规格补充和质量提升**，而非从零创建。
>
> **团队角色分配**：
>
> | 角色 | 职责 | 提出需求来源 |
> |------|------|-------------|
> | 需求分析师（ANALYST） | 读者需求分析、跨角色比对表、优先级矩阵、需求追溯 | review-synthesis R2-4 |
> | 架构顾问（SYSA） | 全局架构依赖图、概念依赖关系梳理 | review-synthesis R2-2 |
> | 前端架构师（FRONTEND） | Docsify 侧边栏导航优化、分页方案、搜索增强 | review-synthesis R2-FE-2 |
> | UI设计师（UX） | 角色自诊交互设计、各章价值声明块规范 | review-synthesis R2-UX |
> | 测试工程师（QA） | 验证标准补充、导航链接有效性检查 | review-synthesis R2-QA-1 |

> **团队分工**：需求分析师主导架构，UI设计师辅助自诊体验设计

- **学习目标**：

- **前置知识**：无（这是全书第一页）

#### 大纲

1. **角色自诊区**
   - 自我诊断问卷（3-5 个 Yes/No 问题 → 推荐对应角色路径）
   - 快速判断流程：我是什么角色 → 我的技术栈 → 我的目标 → 推荐路径

2. **前置知识确认**
   - 必须：至少熟悉一种编程语言
   - 必须：基本的命令行和 Git 使用经验
   - 建议：使用过至少一种 AI 编程助手
   - 可选：了解 Agent/LLM/MCP 基本概念

3. **本书不涵盖的内容**
   - 编程基础语法
   - OpenCode Rust 内部实现
   - 具体云平台（AWS/GCP）完整教程

   - 版本更新说明和 CHANGELOG 链接

#### 核心概念

- **后设导航（Meta-Navigation）**：本章不是书籍的"前言"，而是"书的书"——帮助读者理解整本书的结构，再决定如何阅读。这是全书设计中最容易被忽视但最重要的概念。
- **角色自诊的工程价值**：不是简单的"你是谁"，而是通过结构化问题链引导读者自我定位，避免"觉得跟自己无关就跳过"的损失。

### Article 0.2: 多角色阅读路径

> **团队分工**：需求分析师定义角色路径映射，架构顾问绘制全局依赖图

- **阅读时间**：10 min
- **学习目标**：
  - 理解 13 种读者角色的定义和覆盖范围
  - 根据自身角色找到对应的阅读路径
  - 理解不同路径之间的差异和重叠
- **前置知识**：Article 0.1（已完成角色自诊）

#### 大纲

1. **13 种读者角色分类**
   - 5 核心角色：BEGINNER, POWER, LEAD, SKILL, MANAGER
   - 8 扩展角色：ANALYST, SYSA, BACKEND, FRONTEND, UX, QA, SECURITY, REDTEAM
   - 每个角色的定义、典型背景、核心关注点

2. **各角色的完整阅读路径**
   - 每条路径标注：章节/文章编号、建议用时、阅读模式（精读/浏览/跳过）
   - 与 USER STORY 的追溯映射（哪些用户故事由哪些路径覆盖） **（需求分析师 P1 新增）**
   - 附加资源推荐（先读/后读的外部材料）

3. **全局架构依赖图** **（架构顾问 P0 新增）**
   - 46 篇文章之间的概念依赖关系总图
   - 标注前置章节和后置章节
   - 帮助跳跃阅读者在遇到不理解的概念时快速定位到前驱文章

4. **跨路径对比**
   - 路径重叠区域的标注（哪些章节是所有路径都推荐的）
   - 路径切换指南（阅读中发现当前路径不适合时如何切换）

5. **阅读节奏建议**
   - 各路径的预计阅读时间
   - 每天的推荐阅读量
   - 实操与阅读的时间配比建议

#### 核心概念

- **角色 ≠ 身份**：一个读者可以同时是 BEGINNER（在某些领域）和 POWER（在其他领域）。阅读路径是动态的，可以混合使用。
- **阅读路径即内容设计**：每条路径的可读性直接反映了书籍内容设计的质量——如果一条路径读完出现"跳不过去的断层"，说明内容依赖关系设计有问题。

### Article 0.3: 如何使用本书

> **团队分工**：前端架构师评估 Docsify 导航方案，UI设计师提供阅读体验规范

- **阅读时间**：5 min
- **学习目标**：

  - 了解如何在 GitHub 上反馈和贡献
- **前置知识**：Article 0.2（已选择阅读路径）

#### 大纲

1. **两种阅读模式**
   - 逐章精读（系统型学习者）：Ch0 → Ch1 → ... → Ch7
   - 按需跳跃（目标型读者）：路径驱动，按需回溯
   - 两种模式的选择决策树

2. **阅读技巧**
   - 双窗口配置（一个看书，一个开编辑器实操）
   - 先理解概念再复制配置
   - 跳过不相关的技术栈章节
   - 带着真实项目来读

3. **导航系统说明** **（前端架构师 P1 补充）**
   - Docsify 侧边栏的层级结构和折叠逻辑
   - 46 篇文章的侧边栏分组策略（避免导航过长）
   - 搜索功能的使用技巧（mdBook 内置搜索支持的搜索模式）
   - 页面间导航（上一页/下一页，mdBook 分页导航）

4. **价值声明块规范** **（UI设计师 P1 补充）**
   - 每章开头的标准格式（目标读者/前驱知识/学习收获/预计投入时间）
   - 价值声明块的视觉标注方式

5. **如何参与贡献**
   - GitHub Issues 提交反馈
   - PR 提交流程和规范
   - 贡献者指南链接

#### 核心概念

- **阅读即调试**：技术书籍的阅读是一个反馈循环——读概念 → 实操验证 → 遇到问题 → 回溯解决。好的阅读方式就是让这个循环更短。
