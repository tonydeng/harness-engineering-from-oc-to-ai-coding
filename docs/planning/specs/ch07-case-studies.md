**章节核心主题**：所有概念和最佳实践在真实项目中的端到端验证。

### Article 7.2: 案例二——遗留系统现代化
- **阅读时间**：30 min
- **学习目标**：
  - 掌握遗留系统的全方面审计方法
  - 理解从审计→评估→重构→测试的完整流程
  - 掌握安全审计团队（security-research）的使用
  - 理解增量重构的工程策略
- **前置知识**：Ch6（MCP和Plugin基础），Ch4（Team Mode基础）
- **源材料映射**：OpenCode实战 05（案例二：安全审计现有项目）+ OpenCode实战 05（案例三：从需求到产品全流程）+ OpenCode实战 03（7-Agent Pipeline）

#### 大纲
1. 项目现状
   - 遗留Node.js项目（247文件、186依赖、34K行代码）
   - 已知问题：安全漏洞多、无测试覆盖、依赖过时
   - 改造目标：安全加固 + 测试覆盖 + 依赖升级
2. 阶段一：项目全景扫描
   - /init-deep 深度初始化
   - @explore全量扫描（文件统计/依赖分析/安全扫描）
   - 生成项目现状报告
3. 阶段二：安全审计
   - security-research Team（5人并行审计：3漏洞猎手+2 PoC工程师）
   - 静态分析 + 依赖审计 + 认证审查 + 基础设施审计 + 合规检查
   - 审计报告（CVSS评分+修复优先级）
   - 经典发现示例：SQL注入、XSS、JWT未验证签名
4. 阶段三：重构计划
   - 基于审计报告制定分阶段修复路线图
   - 首期：紧急安全修复
   - 二期：重构 + 测试覆盖
   - 三期：依赖升级
   - 四期：安全体系建立
5. 阶段四：增量改造
   - 7-Agent Pipeline执行安全修复
   - 添加单元测试和集成测试
   - 依赖安全升级
6. 阶段五：验证与交付
   - 全面回归测试
   - 二次安全审计
   - 改造前后对比

#### 核心概念
- **遗留系统改造的策略选择**："改造"不是"重写"——遗留系统的价值在于已验证的业务逻辑。Harness Engineering的策略是先用安全审计摸清现状，再用增量方式逐步改造。
- **审计→计划→执行→验证的闭环**：与从零搭建不同，遗留系统改造需要更多的"审计"环节和更谨慎的"增量"策略。
- **修复路线图的分期设计**：按CVSS评分+业务影响排序，先修最危险和最紧急的。
