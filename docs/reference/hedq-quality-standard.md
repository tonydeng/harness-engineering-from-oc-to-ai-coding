# HEDQ 质量评价标准

> **Harness Engineering Documentation Quality** — 8 维书籍内容质量评价体系
> **版本**: v1.0 | **日期**: 2026-06-28
> **设计依据**: Darwin SKILL 9 维 Rubric (SkillLens arXiv 2605.23899) + ELISA 文档评估 + CNCF TechDocs + AGENTS.md 规范
> **核心对齐**: AGENTS.md（品牌名、链接、代码块、文风、Mermaid 配色）

---

## 一、体系概览

HEDQ 将书籍内容质量分解为 **8 个维度**，总分 **100 分**，三个评估层面：

```
内容层 (45分)      呈现层 (35分)      实用层 (20分)
┌─────────────┐  ┌─────────────┐   ┌─────────────┐
│ D2 准确性 20 │  │ D1 结构  15 │   │ D4 实操  15 │
│ D3 角色   15 │  │ D6 文风  10 │   │ D8 图表   5 │
│ D5 反面   10 │  │ D7 术语  10 │   └─────────────┘
└─────────────┘  └─────────────┘
```

| 维度 | 名称 | 权重 | 自动检测满分 | 评估层 | 自动化率 |
|------|------|:----:|:----------:|:------:|:--------:|
| D1 | 结构与元数据规范性 | 15 | **11**¹ | 呈现层 | ~95% |
| D2 | 内容准确性与时效性 | 20 | **6**² | 内容层 | ~50% |
| D3 | 读者角色覆盖与导航 | 15 | **6**³ | 内容层 | ~40% |
| D4 | 代码块格式与注释覆盖率 | 15 | **4**⁴ | 实用层 | ~60% |
| D5 | 反面案例与边界条件 | 10 | 10 | 内容层 | ~40% |
| D6 | 文风与可读性 | 10 | **2**⁵ | 呈现层 | ~30% |
| D7 | 术语与品牌一致性 | 10 | 10 | 呈现层 | ~95% |
| D8 | 图表与可视化质量 | 5 | **3.5**⁶ | 实用层 | ~40% |
| **合计** | | **100** | **52.5** | | **~55%** |

> **注**：自动检测满分 ≠ 标准定义权重。部分维度的自动化检测仅覆盖其子项的子集：
> ¹ D1 自动化检测覆盖 D1.1（SUMMARY 路径）、D1.2（内部链接）、D1.4（品牌名），未覆盖 D1.3（代码块 `:path` 注释，该检查归入 D4.1）。
> ² D2 自动化检测仅覆盖 D2.2（版本新鲜度），未覆盖 D2.1（技术事实验证，需 deep-research）和 D2.3（源材料映射）。
> ³ D3 自动化检测覆盖 D3.1（追溯矩阵覆盖率），未覆盖 D3.2（阅读路径）和 D3.3（信息分类）。
> ⁴ D4 自动化检测覆盖 D4.1（代码块 `:path` 注释覆盖率），未覆盖 D4.2（配置文件）和 D4.3（步骤重现）。注意：D4.1 检查的是代码块格式注释，而非代码示例是否可运行，完整实操检测需人工介入。
> ⁵ D6 自动化检测覆盖 D6.3（AI 腔检测），未覆盖 D6.1（说人话）、D6.2（段落收益）、D6.4（简洁性）。
> ⁶ D8 自动化检测覆盖 D8.1（Mermaid 语法）和 D8.2（配色合规），未覆盖 D8.3（图表理解辅助性）。

## 二、评分等级

| 等级 | 分数 | 含义 |
|:----:|:----:|------|
| 🟢 **READY** | ≥90 | 可发布，无需修改 |
| 🟡 **CONDITIONAL** | 75–89 | 有条件发布，修复 P1 问题后即可 |
| 🟠 **NEEDS WORK** | 60–74 | 需修改，有 P0/P1 问题 |
| 🔴 **DRAFT** | <60 | 不可发布，需大幅重写 |

### P0 一票降级规则

若任何维度存在 **P0 级违规**（关键事实错误、不存在的 API、无效配置、断裂的核心链接），
则最终评级**强制降一级**：

| 原始评级 | 降级后评级 |
|:--------:|:----------:|
| READY | CONDITIONAL |
| CONDITIONAL | NEEDS WORK |
| NEEDS WORK | DRAFT |
| DRAFT | DRAFT |

此规则确保 P0 问题不会被高分维度掩盖。

## 三、D1 — 结构与元数据规范性（15 分）

### 子项与权重

| 子项 ID | 子项名称 | 满分 | 检查方法 |
|---------|---------|:----:|---------|
| D1.1 | SUMMARY.md 路径完整性 | 3 | 遍历 SUMMARY.md 中所有 `(路径.md)`，检查磁盘文件存在 |
| D1.2 | 正文内部链接有效性 | 4 | 提取全文 `.md` 链接，解析相对路径，检查目标存在 |
| D1.3 | 代码块 `language:path` 格式 | 4 | 非 Mermaid 代码块必须有 `language:相对路径` 格式 |
| D1.4 | 品牌名与规范拼写 | 4 | 检查 OpenCode / oh-my-openagent / MCP / mdBook |

### 扣分规则

| 违规 | 扣分 |
|------|:----:|
| SUMMARY.md 中路径目标不存在 | 每条扣 1 分 |
| 正文内部链接断链 | 每条扣 1 分（同一文件重复不计） |
| 非 Mermaid 代码块缺 `:path` | 每个扣 0.5 分 |
| 品牌名拼写错误 | 每处扣 1 分 |

### 计算方式

```
D1 = 3 × (无缺失路径数 / 总路径数) + 4 × (有效链接数 / 总链接数) +
     4 × (合规代码块数 / 总非 Mermaid 代码块数) + 4 × (正确品牌名数 / 总品牌名出现数)
```

### 打分示例

```text
D1.1: SUMMARY.md 67/67 路径存在  → 3 × 1.0 = 3.0
D1.2: 1092/1094 链接有效 (2 个断链) → 4 × 0.998 = 3.99 ≈ 3.0 (每条断链扣 1 分，封顶至 0)
      → 应用扣分规则: 每断链扣 1 分, 2 条 × 1 = 2.0 → D1.2 = 4.0 - 2.0 = 2.0
D1.3: 3 个代码块缺 path → 3 × 0.5 = -1.5 → 4.0 - 1.5 = 2.5
D1.4: 1 处 "Opencode" → 扣 1 → 4.0 - 1.0 = 3.0

D1 = 3.0 + 2.0 + 2.5 + 3.0 = 10.5 / 15
```

| 子项 ID | 子项名称 | 满分 | 检查方法 |
|---------|---------|:----:|---------|

### 扣分规则

| 违规等级 | 定义 | 扣分 |
|:--------:|------|:----:|
| 🔴 P0 | 关键事实错误（不存在的 API、无效配置） | 子项扣 8 分（归零） |
| 🟡 P1 | 版本过旧、配置参数过时 | 子项扣 4 分 |

```

```

### 打分示例

```text
D2.1: 3 个抽样断言全部通过 deep-research 验证 → 8.0/8
D2.2: 扫描到 "v1.15.x" 但最新为 v1.16.x → P1, 扣 4 → 2.0/6
D2.3: source-cross-refs.md 中 350 个引用路径 全部可访问 → 6.0/6

D2 = 8.0 + 2.0 + 6.0 = 16.0 / 20
```

## 五、D3 — 读者角色覆盖与导航（15 分）

### 子项与权重

| 子项 ID | 子项名称 | 满分 | 检查方法 |
|---------|---------|:----:|---------|
| D3.1 | 用户故事实质覆盖 | 6 | 逐条评估 55 个故事覆盖深度 |
| D3.2 | 阅读路径可操作性 | 5 | 5 个高优角色路径实操验证 |
| D3.3 | 信息类型分类 | 4 | 每篇明确标注概念/操作/参考类型 |

### 覆盖深度评分标准

| 评级 | 定义 | 分值 |
|:----:|------|:----:|
| COMPLETE | 用户故事被实质满足 | 5 |
| SUBSTANTIAL | 映射存在但深度不足 | 3 |
| PARTIAL | 有相关内容但缺口明显 | 1 |
| MISSING | 无任何内容覆盖 | 0 |

### 计算方式

```
故事覆盖率 = Σ(各故事深度分) / (55 × 5)
D3.1 = 6 × 故事覆盖率
D3.2 = 5 × (可操作路径数 / 5)
D3.3 = 4 × (有类型标注篇数 / 总篇数)
D3 = D3.1 + D3.2 + D3.3
```

### 打分示例

```text
D3.1: 14 角色 × 55 故事, 加权覆盖率 = 80%
      D3.1 = 6 × 0.80 = 4.8
D3.2: 5 条阅读路径, 4 条可操作 → 5 × 0.8 = 4.0
D3.3: 56 篇中 48 篇有类型标注 → 4 × 0.857 = 3.4

D3 = 4.8 + 4.0 + 3.4 = 12.2 / 15
```

## 六、D4 — 代码块格式与注释覆盖率（15 分）

### 子项与权重

| 子项 ID | 子项名称 | 满分 | 检查方法 |
|---------|---------|:----:|---------|
| D4.1 | 代码块 `language:path` 注释覆盖率 | 4 | 非白名单代码块必须有 `language:相对路径` 格式（自动化：`checks/d4_codeblocks.py`） |
| D4.2 | 配置文件直接可用 | 5 | examples/ 文件有版本标注 + 说明文档（**未自动化**） |
| D4.3 | 步骤可重现 | 4 | 按步骤式文章操作到预期结果（**未自动化**） |
| D4.4 | 代码示例可运行 | 2 | 提取代码块，逐块执行（子 agent / bash，**未自动化**） |

> **注**：D4 原名"实操可执行性"，但自动化检测仅覆盖 D4.1（代码块 `:path` 注释格式），
> 完整的代码可运行性检测需人工介入。故维度名称调整为"代码块格式与注释覆盖率"以准确反映自动化范围。
> D4.2/D4.3/D4.4 的权重合计 11 分未纳入自动检测，仅 D4.1 的 4 分计入自动检测口径。

### 代码块格式评分

```text
覆盖率 = (有 language:path 注释的代码块数) / (非白名单代码块总数)
D4.1 = max(0, 4 - 缺失数 × 0.3)
```

白名单类型（无需 path 注释）：mermaid、bash、sh、shell、console、text、json、csv、toml、yaml、yml、markdown、dockerfile、html、css。

### 打分示例

```text
D4.1: 120 个非白名单代码块, 3 个缺 :path 注释
      D4.1 = max(0, 4 - 3×0.3) = 3.1
D4.2: examples/ 36 个文件, 25 个有版本标注（人工检查）
      5 × (25/36) = 3.5
D4.3: 5 篇步骤式文章, 4 篇可完整重现（人工检查）
      4 × 0.8 = 3.2
D4.4: 12 个代码块, 10 个可运行, 2 个缺依赖（人工检查）
      2 × (10/12) = 1.7

D4 = 3.1 + 3.5 + 3.2 + 1.7 = 11.5 / 15
（自动检测口径仅 D4.1 = 3.1 / 4）
```

## 八、D6 — 文风与可读性（10 分）

### 子项与权重

| 子项 ID | 子项名称 | 满分 | 检查方法 |
|---------|---------|:----:|---------|
| D6.1 | "说人话"原则 | 3 | 优先具体操作语言，避免抽象隐喻 |
| D6.2 | 每段回答读者问题 | 3 | 每段话有明确读者收益 |

| D6.4 | 简洁性 | 2 | 段落平均长度 ≤ 5 句 |

### 禁用词库（自动检查）

```
说白了|换句话说|首先|其次|再次|综上所述|综上|
值得注意的是|不言而喻|显而易见|众所周知|毫无疑问|
需要注意的是|值得一提的是|在一定程度上|从某种角度来说|
在...方面|在...层面|基于此|因此可以说|不容忽视|
不可否认|不可置否|不得不承认|从本质上讲|从另一个角度
```

> **注**：`显而易见` 使用负向回视断言 `(?<!非)显而易见`，排除合法技术用语"非显而易见"。
> 正常的过渡词（首先、其次、再次）虽在列表中，但 `checks/d6_writing_style.py` 已将其移除，
> 因为这些是中文写作的正常组成部分。

### 扣分规则

```
D6.3: 命中每个禁用词扣 0.2 分, 扣完为止
```

### 打分示例

```text
D6.1: 17 段中 14 段有明确操作语言 → 3 × (14/17) = 2.5
D6.2: 17 段中 14 段有明确读者收益 → 3 × (14/17) = 2.5
D6.3: 0 处命中禁用词 → 2.0
D6.4: 平均 4.2 句/段, 无重复段 → 2.0

D6 = 2.5 + 2.5 + 2.0 + 2.0 = 9.0 / 10
```

## 十一、评分流程

### 重要说明：两种总分口径

本标准定义了两个总分口径：

| 口径 | 满分 | 说明 |
|:----:|:----:|------|
| **标准权重** | 100 | 完整的 8 维质量评估，含人工审核子项。适用于发布评审。 |
| **自动检测** | 52.5 | 仅自动化检测覆盖的子项。`run-hedq.py` 的报告基于此口径。 |

分数换算：`自动检测得分 / 52.5 × 100` 可近似映射到标准权重百分制。

> **注**：自动检测满分 52.5 = D1(11) + D2(6) + D3(6) + D4(4) + D5(10) + D6(2) + D7(10) + D8(3.5)。
> 此前文档曾标注为 72 分，但那是标准权重口径下的部分维度满分累加，与脚本实际输出不符。

### 11.1 快速模式（Phase 1 — 日常 CI）

执行脚本：D1 + D6 + D7

```bash
python scripts/qa/run-hedq.py --quick
```

输出：3 个维度的分数，总计 23 分（自动检测口径：D1=11 + D6=2 + D7=10），耗时 <30 秒。

### 11.2 完整模式（Phase 2 — 发布前）

执行脚本：所有 8 个维度

```bash
python scripts/qa/run-hedq.py
```

输出：8 个维度分数，总计 52.5 分（自动检测口径），耗时 ~30 秒。

### 11.3 汇报输出格式

各脚本统一输出 TSV 格式：

```tsv
dimension	sub_item	score	max	details
D1	D1.1	3.0	3	SUMMARY.md: 67/67 paths ok
D1	D1.2	2.0	4	2 broken links found
...
```

汇总 runner 输出 Markdown 报告：

```markdown
|------|:----:|:----:|---------|

| D2 内容准确性 | 16.0/20 | 🟢 | — |
| ... | ... | ... | ... |
| **总分** | **72.3/100** | 🟠 NEEDS WORK | P0: 2 个断链待修复 |
```
