# 芒格视角审查：Article 1.4 "ecosystem-comparison.md"

**审查者**：查理·芒格（逆向思考 + 多元思维模型）  
**审查日期**：2026-06-06  
**方法**：Inversion——不问"哪个工具最好"，问"哪个工具选错最致命"

---

## 一句话定性

**一篇排版华丽、信息量足的营销材料，伪装成了公平的技术对比。评分系统有四个致命问题，决策树只回答了半个问题。**

---

## 1. 虚假精确（False Precision）：1-5 评分系统

这是最严重的问题。18 款工具 × 8 个维度 × 1-5 分 = 144 个看起来精确的数据点。问题是：

**（a）维度不可公度。** 把"开源性"（二进制属性）和"学习曲线"（主观体验）加起来得到一个总分，跟把苹果的重量加上橘子的价格一样荒谬。OpenCode 总分 36/40——很巧，最高的那个——但这数字没有物理意义。

**（b）评分标准不透明。** "Agent 类型"的评分是一场分类战争：为什么"编排平台=5"而"补全器=2"？这是建立在一个未声明的假设上——Agent 化程度越高越好。但对于一个只想在 VSCode 里自动补全 `for` 循环的开发者，"编排平台"是过度工程。文章自己第 5 节承认 OpenCode "学习曲线陡峭"，但评分时不给惩罚。

**（c）粒度是假的。** "完全开源(5) / 部分开源(3) / 闭源(1)"——如果你真的认真对待开源，它就是 0 或 1。License 能不能商用？社区活不活跃？维护者有几位？这些比 5/3/1 重要得多。

**结论：这组评分给人"量化分析"的幻觉，实际上是一组精心编排的排名。它跟魔术一样有效——因为它也跟魔术一样，你相信了。**（芒格语：The scoring system is raisins mixed with turds. Still turds.）

---

## 2. 可用性偏误（Availability Bias）导致的比较陷阱

文章列出了 18 款工具，但对照组的完整性有问题：

- **OpenAI Codex** ——"已下线"，占了一个对比位。拿一个不存在的产品来填充矩阵，唯一的作用是让还在运营的工具看起来更好。
- **ChatGPT Code Interpreter** ——严格来说不是"AI 编程工具"，是数据分析沙箱。拉它进来也是为了凑数。
- 真正值得对比的维度反而被忽略：**更新频率**、**社区活跃度**、**Issue 响应时间**、**重大故障历史**——这些是 inversion 最关心的"什么东西会搞砸"的数据。

**Inversion 追问**：如果 18 款中有 3 款在 12 个月内会停止维护或改变商业模式，读者能看出哪几款最危险吗？从这篇文章里看不出。

## 3. 决策树只问了一半

决策树（3.1 节）的路径是"你想要什么？"——这是正向思考，不是逆向思考。

**缺失的逆向分支（Inversion branches）：**

| 正向路径 | 缺位的逆向问题 |
|----------|---------------|
| "合规严格 → OpenCode 自托管" | **"你们团队有没有人运维自托管的服务？没有的话谁负责修？"** |
| "预算有限 → Continue + Tabby" | **"免费工具没人付工资给维护者，哪天不更新了你们换什么？"** |
| "快速上手 → Cursor" | **"Cursor 被收购了你的代码历史和配置怎么迁移？"** |
| "GitHub 生态 → Copilot" | **"微软哪天把 Copilot 集成进 Visual Studio 独家功能，你的团队怎么应对？"** |

**建议增加一个"风险决策树"**：先问"什么最可能让你的选型决策失败"，过滤掉高风险选项，再在安全的剩余选项中做选择——这是标准的逆向工作流。

---

## 4. Lollapalooza 效应：多重偏误叠加导致读者高估开源的普适性

文章的立场是亲开源的，这不一定是错的，但多重叙事手法叠加放大了这种倾向：

| 偏误 | 表现 |
|------|------|
| **可得性** | OpenCode 独享整节"优势与局限"讨论 |
| **确认偏误** | 评分体系的结构性偏好（开源=5, 闭源=1）反复强化"开源=好" |
| **现状偏误** | 决策树的枝桠最终都导向开源选项（9/10 叶子节点是开源工具） |
| **社会认同** | （18 款对比的权威姿态）让读者觉得这是一份客观行业分析而非立场表达 |

这些偏误单独看都不致命，但叠加在一起——Lollapalooza——产生了一个效果：**读者可能得出结论"不开源的工具都不值得认真考虑"，但这个结论在文章里从未被论证过。**

> 注意：这版书是开源书籍，亲开源立场合乎情理。问题是文章**看起来**像中立对比，但**实质上**是一份精心构建的开源论证。这种"声称中立、实际立场"的文体本身就是一种误导。

## 5. 什么是真正该问的问题（Inversion 版选型框架）

文章的核心问题："哪款工具最适合我的场景？"

**芒格版问题**："哪款工具选错了会让我在接下来 18 个月里最痛苦？"

由此推导出文章缺失的——或者说权重不足的——五个维度的评价：

1. **切换成本（高于一切）** ——工具 A 比工具 B 好 20%，但如果换工具要花两周重搭配置工作流，那 20% 的优势是幻觉。文章对此零讨论。
2. **维护者风险** ——开源不等于无风险。一个只有 3 个 committer 的开源项目比一个闭源但有 500 人团队的产品更危险。文章把"开源"等同于"安全"，这是 naive。
3. **锁定形态** ——供应商锁定不止一种：有 API 锁定（只能用特定模型）、有数据锁定（配置和历史记录迁不走）、有工作流锁定（肌肉记忆绑死在一个编辑器里）。文章只讨论了第一种。
4. **降级路径** ——如果选了 OpenCode 但团队用不来，能不能降级到 Cursor？如果能，代价是什么？文章从不同时推荐 Plan B。
5. **工具消亡** ——OpenAI Codex 已经下架了，Google 砍掉的产品比大多数公司的产品总数还多。文章零风险分析。

## 附录：芒格式的快速检验

把 144 个评分数据点压缩成 3 个真正有用的判断：

1. **如果你没人运维服务器 → 不自托管**（删掉 Continue/Tabby/OpenCode 自托管选项）
2. **如果你无法忍受大版本不兼容 → 选成熟的闭源产品**（开源项目 break change 频率更高）
3. **如果你在意长期稳定性 → 看 commit 频率和创始团队背景，而不是评分**

> **"All I want to know is where I'm going to die, so I'll never go there."**  
> 这篇文章的回答是：你 die 的地方是 **选了一个自己没能力运维的工具**，以及 **被总分误导选错了方向**。两件事它都没帮你避免。
