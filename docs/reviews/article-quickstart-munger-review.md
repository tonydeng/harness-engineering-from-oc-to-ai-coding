## 1. 激励偏差（Incentive Bias）— 最大问题

**核心问题**：文章中"OpenCode Zen"被推荐为"新手推荐"方案，但 OpenCode Zen 是 **OpenCode 团队自身提供的商业服务**。文中没有充分披露这一利益冲突。

### 具体证据

**行 28-29**：
> OpenCode Zen 账户（推荐新手，在 https://opencode.ai/auth 注册）

**行 193-194**（标题）：
> 方式一：OpenCode Zen（推荐新手）

**行 229-233**（优势列表）：
> - 无需分别注册多个 Provider
> - 模型已经过 OpenCode 团队验证
> - 统一计费，简化管理
> - 支持多种高质量模型

**行 373-377**（对比表）：
> | **OpenCode Zen** | 新手、想省心的用户 | 一站式、已验证模型、统一计费 | 需要注册 OpenCode 账户 |

### Munger 诊断

**这是经典的"谁付费，谁说话"偏差（Incentive-Caused Bias）**：推荐自家产品本身不违法，但问题在于：

1. **未充分披露利益冲突**：文章从头到尾没说"OpenCode Zen 由 OpenCode 团队运营"或"作者与 OpenCode 存在关联"。对于一个声称中立的技术书籍，这违反了"披露第一"原则。

2. **劣势列刻意弱化**：对比表（行 373-377）中，OpenCode Zen 的"劣势"只写了"需要注册 OpenCode 账户"。它没有说：**你被锁定在该平台的计费体系中；你无法直接控制使用的底层模型；你的使用数据经过 OpenCode 的中间层；当 OpenCode Zen 出现故障时你无法切换到备用 Provider（除非同时配置多个）。**

3. **排序带来的锚定效应**：三种配置方式中，OpenCode Zen 排第一，且冠以"推荐新手"标签。新手读者最容易被锚定在这个选项上，而不会认真对比成本和灵活性。

4. **选择性呈现"优势"**：OpenCode Zen 的"统一计费"被列为优势，但同一篇文章中并没有给出任何价格信息。读者无法判断"统一计费"是贵还是便宜。

### 建议

- 在推荐 OpenCode Zen 前，添加利益冲突声明（如"OpenCode Zen 由 OpenCode 团队运营，本文作者与之存在合作关系"）
- 增加成本透明性：至少给出 OpenCode Zen 的价格范围，与直接使用 API 做对比
- 在对比表劣势列中加入"供应商锁定"、"数据经中间层"、"无法直接控制底层模型版本"

## 2. 隐藏成本问题（Hidden Costs）

Munger 原则：**"Never buy anything without knowing the total cost of ownership."**

### 2.1 OpenCode Zen 需要"填写账单信息"但不提价格

**行 206**：
> 浏览器会自动打开 opencode.ai/auth，完成登录并填写账单信息

这是整篇文章中对 OpenCode Zen 价格的**唯一提及**。读者直到需要填写信用卡时才会看到价格，这是一种典型的价格信息不对称策略。

### 2.2 GitHub Copilot 的"AI Credits"信息不完整

**行 313**：
> 自 2026 年 6 月起，GitHub Copilot 采用基于用量的 AI Credits 计费模式。Copilot Pro 每月包含 $15 的 AI Credits，超出部分需额外付费（约 $0.01/credit）。

缺少关键信息：
- OpenCode 使用 Copilot 时，Credits 消耗速率如何？
- 与直接买 API Key 相比，Copilot 代理模式是否有额外溢价？

### 2.3 完全没有总拥有成本（TCO）对比

三种配置方式在同一个表格里，但没有 **价格列**。读者无法回答一个基本问题："如果我每天用 2 小时，一个月要花多少钱？"

### 2.4 time-as-cost 被低估

**行 3-7** 的"预计 20-30 分钟"排除了：
- 首次注册 API 账户（实际常需 15-30 分钟）
- 安装 Node.js（如果尚未安装）
- 安装后遇到网络问题时的排查时间
- 安全配置的理解和学习时间

这是一种 **计划谬误（Planning Fallacy）**。

## 3. 遗漏的最坏情况（Omitted Worst Case Scenarios）

### 3.1 `curl | bash` — 安全反模式

**行 80, 99, 132**：
> curl -fsSL https://opencode.ai/install | bash

文章中没有**任何安全警告**，如：
- 建议先下载审查脚本内容
- 建议验证 checksum
- 说明此方式的信任假设

对于一本自称"Harness Engineering"、第一章原则就是"可控"的书籍，这个遗漏特别讽刺。

### 3.2 AGENTS.md 可能出错

**行 397-398** 的"出生证明"隐喻暗示它是权威的、正确的。但 `/init` 的分析结果可能：
- 误解项目结构
- 错误猜测技术栈
- 遗漏重要的自定义配置

没有任何警示或验证步骤建议。

### 3.3 "Always"权限选项的危险

**行 659-661** 的"Always"选项，新手可能为了省事全部选 "Always"，然后在不知情的情况下让 Agent 获得完全控制权。

## 4. 确认偏误（Confirmation Bias）

**核心问题**：文章只呈现了"使用 OpenCode 的好处"这一面，完全没有讨论其局限性和适用边界。

- **未曾提及"何时不该使用 OpenCode"**
- **75+ Provider 对新手是信息过载而非优势**
- **小结部分只总结"学到了什么"，不总结"需要注意什么"**

## 5. 隐含假设（Unstated Assumptions）

| # | 假设 | 隐含于 | 风险 |
|---|------|--------|------|
| 1 | 读者已有 Node.js 18+ | 行 17-20 | 可能没有 |
| 2 | 读者使用现代终端 | 行 22-25 | 大量 Windows 用户仍在使用 cmd.exe |
| 3 | 读者有 GitHub 账户 | 行 310-362 | 注册+配置时间被忽略 |
| 4 | 读者信任 `curl \| bash` | 行 80 | 安全反模式 |
| 5 | 读者有可供初始化的项目 | 行 389-398 | 新手可能没有 |
| 6 | 读者能阅读英文命令行输出 | 全文 | OpenCode TUI 输出为英文 |
| 7 | 读者理解"Provider"、"Session"等术语 | 行 7 | 新手可能被术语淹没 |

## 6. 叙事谬误（Narrative Fallacy）

**行 398, 677** 将 AGENTS.md 比喻为"出生证明"：
- 暗示一次生成、永久有效 — 实际上项目演进而 AGENTS.md 不会自动更新
- 暗示官方性和完整性 — 实际上 /init 可能不完全正确
- 读者可能不会再次运行 `/init` 来更新

---

## 7. 安全配置的结构性问题

1. **安全配置放在快速上手的末尾**（第 21/24 节），新手可能在配置完所有东西后才看到
2. **"Always"选项的安全幻觉**：新手频繁选 "Always" 等于给 Agent 完全信任
3. **权限模型依赖于用户的持续警醒**：这是最脆弱的安全模型

---

## 8. 锚定效应（Anchoring）

文章将 OpenCode Zen 放在"方式一"位置，并用"推荐新手"标签锚定读者选择。对比表排序也是 OpenCode Zen → 自有 API Key → GitHub Copilot，并非中立。

---

## 总结：最大的三个陷阱

| 风险等级 | 问题 | 读者可能遭受的后果 |
|----------|------|-------------------|
| 🔴 **最高** | 激励偏差：推荐自家商业产品而不充分披露利益冲突 | 做出非最优的经济决策，被锁定在单一供应商 |
| 🔴 **高** | 隐藏成本：OpenCode Zen 的价格需要填信用卡后才看到 | 完成注册后才意识到成本超出预期 |
| 🟡 **中** | 安全反模式：`curl \| bash` 无警告 + "Always"权限选项无说明 | 在首次使用时就建立不安全的工作习惯 |
