# Karpathy 式审阅：`ecosystem-comparison.md` — 工程现实主义视角

> 以 Andrej Karpathy 的 engineering realism 框架审阅本文。
> 审阅者：Karpathy 思维模型（基于公开言论推断，非本人）

---

先说结论：这篇文章做了一件很困难的事——横向比较 18 个工具，覆盖定价、开源性、Agent 能力、隐私保护这么多维度。对于一本入门书来说，它的信息量是充足的，场景化推荐部分（决策树）是真正有用的。

但问题在于：**它把一套主观打分系统包装成了客观对比，而真正的工程判断力恰恰被这个系统遮蔽了。**

---

## 1. 18 个工具的并排放置：一个分类学问题

文章把 18 个工具放在同一个 8 维矩阵里评分。但这里面有些工具是不该放在一起比较的。

Tabby（自托管补全器）和 OpenCode（Agent 编排平台）和 ChatGPT Code Interpreter（数据分析沙箱）——这三个服务的是完全不同的需求。用同一个评分体系去衡量它们，就像拿汽车、自行车和船比谁的游泳能力强。

| 工具 | 实际定位 | 8 维评分是否适用？ |
|------|---------|-------------------|
| Tabby | 补全后端 | 你在评它的 "Agent 类型"？ |
| Code Interpreter | 沙箱执行 | 你在评它的 "开源性"？ |
| OpenAI Codex | 已下线产品 | 为什么还在评分表里？ |

一个更诚实的做法是：**先分 bucket，再在 bucket 内比较。** Bucket A：IDE 内嵌补全（Tabby, Copilot base）；Bucket B：对话式 Agent（Cursor, Windsurf, Continue）；Bucket C：终端 Agent（Claude Code, Aider, Codex CLI）；Bucket D：编排平台（OpenCode, Goose）。跨 bucket 的比较可以用文字描述，而不是用一个统一的数字评分表。

## 2. 评分系统：1-5 的主观性被 "量化" 伪装了

这个打分系统 — 8 个维度，每个 1-5，总分 — 是整篇文章最大的工程现实问题。

让我挑几个：

- **OpenCode 学习曲线 = 2**，Claude Code 学习曲线 = 4。依据是什么？Claude Code 是一个纯终端工具，需要一个有 CLI 偏好的用户才能用好。OpenCode 是终端工具 + 概念体系（Agent/Skill/Workflow/Plugin/MCP/Constraint — 6 个核心概念）。但 "2 vs 4" 的差距暗示 OpenCode 的学习成本是 Claude Code 的两倍多。这个数字有测量依据吗？没有。它是主观分类。

- **OpenCode Provider 自由度 = 5**（75+ Provider）。75 个 Provider 里有几个是生产可用的？它们在 Token 成本、延迟、Code 生成质量上的分布是什么样的？"75+" 这个数字本身没有任何工程意义。我管一个叫 "provider" 的东西不代表它有用。

- **Copilot 隐私 = 1**（云端处理）。Copilot Enterprise 有私有部署方案吗？有代码排除策略吗？实际产品有灰度。1 分等于 "不可用"，但它确实被大规模企业使用。这里评分和现实有 gap。

最关键的问题：**总分排名把文章的核心论点给暴露了。**

| 排名 | 工具 | 总分 |
|------|------|------|
| 1 | OpenCode | 36 |
| 2 | Goose | 31 |
| 3 | Continue | 30 |

这是一个很干净的叙事：OpenCode 遥遥领先，第二名差了 5 分。如果 OpenCode 的学习曲线被扣到 "2" 还能拿 36，那这个评分系统事实上对这本书的主推工具是有利的。

我不是在说评分不诚实。我是说：**当评分系统天然有利于你想要证明的论点时，你要额外怀疑它。** 这是 engineering 基本训练——当你对代码有 bias 时，测试用例要更严格，不是更宽松。

## 3. "Agent 化是确定性方向" — 这个 claim 需要检查

> "Agent 化（从补全到 Agent 执行是确定性方向）"

这个说法我在 2025 到 2026 年听过很多次。

但 "确定性方向" 意味着什么？如果你在 2025 年 2 月全仓押注 Agent 自主写 PR，你会失望。Claude Code 刚发布时连 `git commit` 都要权限确认，Cline 的自主模式经常跑偏，Agent 写测试然后修改代码导致无限循环——这些我都见过。我在 2025 年的评价是："The models are not there. It's slop."

**方向是真实的，但 "确定性" 这个词太重了。** 它会引导读者以为 Agent 编程已经是生产就绪的成熟模式。实际上，Agent 编程的 "march of nines" 爬坡才刚刚开始——从 Demo 里的 90% 成功到企业部署需要的 99.9%，这个 gap 可能还需要 2-3 年。

建议改为："Agent 化是最重要的趋势，但可靠性仍处于早期阶段"。

## 4. March of Nines 思维的缺失 — 文章最大的盲区

这篇文章对每个工具做了漂亮的分类和评分。**但它几乎完全不讨论这些工具在实际部署中可靠性的问题。**

一个工程导向的对比应该问：

- 这些工具在 CI/CD pipeline 中的行为可以预测吗？
- 当模型 API 挂掉时，工具怎么降级？
- 代码生成失败的模式是什么？是无声地产生错误代码，还是给出清晰的错误提示？
- 增量采用 —— 能否从补全起步，逐步过渡到 Agent 执行？
- 错误归因 —— Agent 决策错了，你能通过日志追溯到哪个具体步骤出问题了吗？

这些问题没有被讨论。替代它们的是 "Plugin/扩展能力" 和 "定价模式" ——这些信息当然有用，但它们不回答一个工程师最关心的问题："这个工具在我每天 8 小时的工作中，到底靠不靠谱？"

**文章把工具对比做成了一个 "选美比赛"（哪个工具 feature 多），而不是 "可靠性测试"（哪个工具在真实工作流里炸得少）。** 对于一本讲 "Harness Engineering" 的书来说，这有点矛盾。

## 5. 定价对比 — 事实性数据的易碎性

文章提供了一个详细的定价表格（133-147 行），并注释 "写书时 2026 年 6 月数据"。

工程现实主义看待定价数据的方式是：

- **这些数字在书出版时很可能已经变了。** 文章自己都说了 Copilot 在 2026 年 4 月刚改了计费模式（从订阅到用量计费）。
- **"免费" 和 "付费" 的界定越来越模糊。** Copilot 的 "Pro $10/月" 和它的实际 Token 上限有关；Windsurf 的免费层和 Pro 的差异体现在 Flow Action 次数。单纯的订阅价格不能反映实际使用成本。
- **真正的成本是 Token 消耗 + 调试时间，不是订阅费。** 一个每月 $20 但经常给你生成错误代码、让你花 30 分钟修正的工具，比一个 $10 但不犯错的工具贵得多。

如果是为了给读者一个初始印象，价格对比是 OK 的。不要让它看起来像是选型的主要依据。

## 6. 象限图（75-102 行）：精确坐标的假象

OpenCode 被放在 [0.9, 0.95] 的位置。

我对这个图的反应和评分表一样：这些坐标值怎么来的？[0.9, 0.95] 和 [0.85, 0.75] 的区别在工程上可测量吗？不能。

象限图作为一种视觉沟通工具是好的——它帮助读者快速理解工具在 "自主性 × 扩展性" 空间里的相对位置。但不要把它当作一个精确的定位工具。我建议：

1. 放弃精确坐标，改用区间（"高自主性、高扩展性" vs "中自主性、低扩展性"）
2. 或者明确注释：坐标是示意性的，基于作者判断

## 7. 做得对的部分

说了这么多问题，我要专门表扬一下文章真正做对的部分。

**决策树（160-203 行）是全文最诚实的部分。** 它没有假装自己是客观的、数据驱动的。它就是 "if this, then try that"——这是工程问题的正确应对方式。不同场景 -> 不同约束 -> 不同推荐。没有分数，没有象限图，就是判断路径。这部分保留就好。

**场景化推荐速查表（207-218 行）** 同样实用。个人开发者、小团队、企业各自的需求被细化到具体 tool 建议，并且给出了理由。这比评分表有用得多。

**开源 vs 闭源框架（224-257 行）** 也是好的。它抓住了开源选型最根本的三个约束条件：合规、定制需求、运维能力。这个决策框架是工程思维，不是营销思维。

**坦白定价数据限度（52, 148 行）**—— "以当前实际定价为准" 的注释虽然简短，但态度对。

## 具体改进建议

### 建议一：砍掉评分表，或者给它加上清晰的主观性标注

把 106-125 行的详细评分表替换或补充为：
- **分 bucket 比较**：同一类别的工具之间做差异化分析，而不是 18 个工具用同一把尺
- **真实可靠性信息**：哪个工具有公开的故障报告？哪个工具的 Agent 模式有回滚机制？
- 如果一定要保留评分，用一个明显的 disclaimer：**"以下评分为作者判断，非定量测量"**

### 建议二：Agent 化趋势加 "但可靠性不足" 的补充

在 "Agent 化是确定性方向" 后面直接补一段：
```
但注意：当前 Agent 模式的可靠性远未成熟。
2025 年底的 Agent 产品在复杂任务上的成功率（一次通过）不到 50%。
真正的 Agent 工程化还需要 1-2 年的 march of nines 爬坡。
选型时优先选择可逐步切换（人工确认→半自动→全自动）的工具。
```

### 建议三：象限图改用粗略区间

[0.9, 0.95] 改为 "高自主性，高扩展性" 的描述性标签。或者加注释说明坐标意义。

### 建议四：增加 "工具退化" 的讨论

> 工程现实：没有工具永远变好。Copilot 在 2025-2026 年经历了多次质量下降（模型切换、策略调整）。闭源工具的功能可能随时被移除。开源工具可能因为维护者疲劳而停滞。

这是一个真实工程问题 ——「工具选择」是一个动态决策，不是一次性决策。文章把它写成静态快照了。

---

## 总结

这篇文章最大的价值在于它的场景化推荐、决策树和开源 vs 闭源框架。这些是真正的工程思维——从约束出发，而非从 feature 出发。

但它最大的问题在于 8 维评分系统和象限图。这些部分把主观判断包装成客观数据，并且评分结构恰好有利于这本书记述的主要工具。对于一个自称 "工程化" 的书籍来说，这不是一个好信号。

Imo，好的工具对比应该告诉你两件事：(1) 这个工具擅长的场景是什么，(2) 它在哪些场景下会炸。这篇文章做了第一件事的大部分，但几乎没有碰第二件事。

一个简单的改进方向：**去掉总分排名，把评分改成范围估计（"学习曲线：中-高" 而非 "2/5"），增加每个工具已知的失败模式。** 这样不会增加太多篇幅，但工程诚实度会有一个跳跃。

就这样。
