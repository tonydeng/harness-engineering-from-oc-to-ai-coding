## 1. 读者角色缺失分析

### 1.1 现有的角色盲区

当前 PRD §5 和 User Stories 定义的 5 个角色（入门开发者/效率开发者/技术负责人/Skill 作者/工程经理），全部围绕 **"读者需要什么内容"** 来定义。没有一个角色关心 **"内容如何呈现给读者"**。

对于一本包含 **30+ 张图表、22+ 篇文章（每篇 >200 行）、7 章结构** 的技术书籍，呈现质量和阅读体验将直接影响所有角色的阅读效率和理解度。

### 1.2 新增角色：文档UX专家 / 技术文档读者体验顾问

| 字段 | 定义 |
|------|------|
| **角色标识** | US-UX |
| **角色名称** | 文档UX专家 / 技术文档读者体验顾问 |
| **背景** | 有技术文档编写或信息架构经验，关注知识传播效率。可能是有设计背景的技术写作者，或是负责开发者文档的 Developer Advocate。对 Docsify/MkDocs/GitBook 等文档工具、Markdown 最佳实践、WCAG 可访问性标准有了解。 |
| **核心目标** | 确保所有读者角色能以最低认知负荷、最高信息获取效率阅读本书内容。在桌面和移动端都能获得一致、无障碍的阅读体验。 |
| **关注点** | - 信息层级与视觉节奏：标题深度、内容分段、概念/示例/最佳实践的视觉区分<br>- 图表可读性：Mermaid 颜色方案、布局一致性、SVG 导出质量<br>- 代码块可读性：文件名标注、行高亮、拷贝按钮<br>- 可访问性：色彩对比度、屏幕阅读器支持、Alt 文本<br>- 移动端适配：表格滚动、图表缩放、侧边栏折叠<br>- 导航系统：跨文章链接、面包屑、进度指示<br>- 中文字体渲染：合适的正文字体/字号/行高<br>- 暗色模式兼容 |
| **推荐阅读路径** | 全书通读（重点关注：侧边栏结构、跨章节链接、图表一致性、代码块标注、移动端效果） |
| **预计目标占比** | —（属于质量保障角色，不独立贡献占比；其关注点影响所有角色） |

## 2. 用户故事（UX 视角）

### US-UX-01：一致且可读的图表体验

> **作为** 一个文档UX专家，  
> **我想要** 全书所有 Mermaid 图表使用统一的色彩方案、布局风格和标注规范，  
> **以便** 读者在任何章节中都能无障碍地阅读图表信息，无需适应不同风格。

**验收标准：**

- [ ] 定义 Mermaid 颜色主题变量（主色/辅色/背景色/线条色），确保与 Docsify 主题色协调
- [ ] 所有流程图（flowchart）统一使用 TB（top-to-bottom）或 LR（left-to-right）方向，全书记录一致的朝向选择
- [ ] 所有时序图（sequenceDiagram）统一参与者命名规范（中文标签/英文ID 的映射规则）
- [ ] 图表中的中文文本设置合适的字体大小，避免在小屏设备上文字重叠
- [ ] 30+ 张图表均附带 Alt 文本描述（Markdown 图片语法 `![描述文字](url)`），便于屏幕阅读器和搜索索引
- [ ] 复杂图表（>10 个节点）提供"点击放大"或"在新标签打开"的能力（mdBook 需配置图片缩放功能）
- [ ] 同一语义元素在不同图表中使用相同颜色（例如 Agent 类节点全书记为同色，Skill 类节点为另一种同色）

**优先级：** P1 | **工作量：** M | **影响角色：** 全部读者

---

### US-UX-02：信息层次清晰的代码体验

> **作为** 一个文档UX专家，  
> **我想要** 所有代码块都有明确的上下文说明（文件名标注、语言高亮、关键行注释），  
> **以便** 读者在复制粘贴时清楚知道"这段代码要放在哪里"和"这段代码的作用是什么"。

**验收标准：**

- [ ] 每条代码块第一行标注文件路径（如 ````json .opencode/opencode.json`），格式全书记统一
- [ ] 使用行内注释 `// <-- 重点行` 或行号范围标注（`` ```json {4-6} ``）标识关键配置行
- [ ] 50+ 个代码示例使用正确的 Prism 语言标识符（bash/json/yaml/markdown/diff）
- [ ] 需要读者修改的部分使用占位符 `<your-api-key>` 并用斜体或高亮标记
- [ ] 长代码块（>20 行）提供"复制"按钮（mdBook 需配置代码复制按钮）
- [ ] diff 格式的代码使用 ````diff` 标注，以红绿对比色展示增删内容
- [ ] 每条代码块前有 1-2 句文字说明"这段代码的用途"

**优先级：** P1 | **工作量：** M | **影响角色：** 全部读者

---

### US-UX-03：移动端无碍阅读

> **作为** 一个文档UX专家，  
> **我想要** 在手机浏览器上阅读时有完整的侧边栏导航、可滚动的代码块和可缩放的图表，  
> **以便** 在通勤/会议间隙也能无障碍地阅读本书内容。

**验收标准：**

- [ ] 在 375px（iPhone SE）和 414px（iPhone Plus）宽度下，侧边栏可正常展开/收起
- [ ] 代码块在移动端可横向滚动（overflow-x: auto），不截断内容
- [ ] Mermaid 图表在移动端自动缩放以适应屏幕宽度，且文字不重叠
- [ ] 表格在移动端可横向滚动（Prd §5 的读者画像表、§3.2 的来源映射表均 >4 列）
- [ ] 中文字体在 16px 基准字号下清晰可读，行高 ≥ 1.8
- [ ] 触控目标（侧边栏按钮/链接）尺寸 ≥ 44x44px（WCAG 2.5.5）
- [ ] 页面加载时无明显布局偏移（CLS < 0.1）

**验收工具：** Chrome DevTools 设备模拟 / 实际手机访问 GitHub Pages 预览

**优先级：** P2 | **工作量：** S | **影响角色：** 初级开发者、效率开发者（移动端阅读频率最高的群体）

## 3. 视觉呈现与阅读体验改进建议

### 3.1 急需：mdBook 配置增强

| 当前问题 | 严重程度 | 建议修复 |
|---------|---------|---------|
| Mermaid 插件未配置 | 🔴 阻塞 | 在 `book.toml` 中配置 `[preprocessor.mermaid]`；否则 30+ 图表全部无法渲染 |
| 无自定义 CSS | 🟡 重要 | 添加 `theme/index.hbs` 和 `theme/css/custom.css`；覆盖默认主题以优化中文字体 |
| 无代码拷贝按钮 | 🟡 重要 | mdBook 默认提供代码复制按钮 |
| 无图片缩放能力 | 🟢 建议 | 配置 mdBook 图片缩放功能（Mermaid 截图后需要） |
| 无分页导航 | 🔵 提示 | mdBook 默认提供分页导航 |

### 3.2 Mermaid 图表设计规范（建议）

针对 30+ 张图表的一致性和可读性，建议在 `theme/css/custom.css` 中定义以下 Mermaid 主题变量：

```
颜色体系：
  - primaryColor:     #4A90D9 (蓝，用于 Agent/系统组件)
  - secondaryColor:   #50C878 (绿，用于 Skill/能力模块)
  - tertiaryColor:    #FF9F43 (橙，用于 Workflow/流程节点)
  - quaternaryColor:  #A66CFF (紫，用于 MCP/外部集成)
  - mainBkg:          #F8FAFC (浅灰背景)
  - lineColor:        #94A3B8 (中性线条)
  - textColor:        #1E293B (深灰正文)
```

布局规范：
- 流程图：统一使用 TB（top-to-bottom）方向
- 时序图：参与者按"触发者→执行者→被调用者"从左到右排列
- 类图/架构图：如节点数 >8，分模块用子图 subgraph 分组

### 3.3 代码块标注规范

建议全书记录以下格式：

```markdown
<!-- 标准格式 -->
```json:.opencode/opencode.json
{
  "key": "value"
}
```

```diff:src/config.ts {3-5}
```

**规则：**
1. 语言标识符后紧跟 `:` + 文件路径
2. 需要高亮的行号范围放在末尾花括号 `{3-5}` 中
3. 配置文件路径优先用相对路径 + 文件全名
4. 命令行输出/交互式操作使用 ````ansi` 或 ````text`

### 3.4 信息层级建议

200+ 行/篇的文章需要清晰的视觉节奏：

```
# H1 标题（仅用于文章标题）
  ↓
## H2 小节标题（用于概念/步骤/示例的分组）
  ↓
概念介绍段落（1-3段，约100-200字）
  ↓
> **示例：** xxx（带背景色的引用块区分示例）
  ↓
代码块
  ↓
> **最佳实践：** xxx（另一种视觉样式的引用块）
  ↓
### H3 子标题（仅在需要细分子主题时使用）
```

建议为"示例"和"最佳实践"使用不同的引用块样式（两种颜色），帮助读者在扫读时快速定位内容类型。

### 3.5 中文字体与排版

| 参数 | 建议值 | 理由 |
|------|--------|------|
| 正文字体 | `system-ui, -apple-system, "PingFang SC", "Noto Sans SC", sans-serif` | 原生中文字体渲染最佳 |
| 正文字号 | 16px（桌面）/ 15px（移动端） | 中文阅读舒适区间 |
| 行高 | 1.8 | 长篇技术阅读最低要求 |
| 代码块字号 | 14px（桌面）/ 13px（移动端） | 代码紧凑但可读 |

### 3.6 导航与信息查找

- **跨章节引用**：建议定义统一格式 `→ [§2.1 Agent 编排](02-core-concepts/agent-orchestration.md)`；全文 6 个核心概念会反复跨章节引用，统一的引用格式降低查找成本
- **术语首次出现**：每个术语首次出现时高亮 + 链接到附录术语表；例如 `**Agent**（→ 见术语表）`
- **文章顶部 YAML frontmatter**：建议扩展字段包含 `tags:`（标签数组），便于 mdBook search 插件索引

### 4.2 §4.2 内容格式规范 补充

当前 4.2 节内容过于简略（仅 5 条规则），建议扩展为以下完整规范：

```
### 4.2 内容格式规范

#### 4.2.1 文档元数据
- 使用 YAML frontmatter 管理文档元数据，必须包含：title（标题）、date（最后更新日期）、tags（标签数组）

## 6. 总结

当前 PRD 和 User Stories 在内容规划上已相当完整，但 **呈现层几乎是一片空白**。对于一个 30+ 图、22+ 篇长文的技术书籍，没有 Mermaid 渲染插件、没有统一图记颜色方案、没有代码标注规范、没有移动端验证标准，这些问题会直接影响所有角色的实际阅读体验。

最紧急的修复是 **在 book.toml 中配置 mdbook-mermaid 预处理器**（否则所有图表在 GitHub Pages 上都是空白），其次是 **编写自定义 CSS 优化中文字体渲染和 Mermaid 配色**。建议将上述补充内容整合到 PRD §4 和 §7 中，确保在进入内容编写阶段前，呈现层的技术基座已经就位。
