## 零、免责声明

我以芒格视角来分析，基于公开言论推断，不是本人观点。

先说结论再展开：**这篇文章犯了一个聪明人常犯的错误——把好故事当成了好论证。** 三个浪潮的叙事、五个瓶颈、三大原则、五大能力——结构漂亮，逻辑工整。但一个结构工整的东西不一定是真的。漂亮结构最容易让你放松警惕，忘记问最基本的问题：**这个框架如果不是真的，文章会少了什么？**

## 一、逆向思考测试（Inversion）

> **不问"这个框架哪里好"，问"如果这个框架是错的，文章哪些地方会不同"**

### Pass / Fail 判断标准

- **Pass** = 文章包含了足以让读者自行推翻框架的信息
- **Fail** = 文章的信息结构使读者更难以发现框架的问题

### 发现

**Fail 1：文章从未考虑过"Harness Engineering 可能是错的"这种可能性。** 第 15-19 行的三次浪潮叙述是一种**辉格史观**（Whig history）——把历史描述为一路进步到今天。如果倒过来看：

- 第一次浪潮的"Prompt Engineering"被描述为原始阶段
- 第二次浪潮的"Context Engineering"（Cursor/Claude Code）被描述为过渡阶段
- 第三次浪潮的"Harness Engineering"被描述为目标/终点

这种叙事的问题是：**它把当前最流行的概念放在了历史终点的位置。** 如果倒过来，Harness Engineering 可能只是第四次浪潮的前奏，届时更好的模型让大部分 Harness 层变得多余。文章没有考虑这个可能性。

**Fail 2：第 208-215 行的 Hashimoto 定义被当作权威背书，但逆向看：** 一个概念诞生不到 4 个月就被写入教科书式的框架——这不是验证的信号，这是营销的信号。价值投资的基本原则是：一个投资标的在被发现后的前 6 个月，你不知道它是被发现了还是被制造了出来。同样的逻辑适用于概念：Harness Engineer 是被"发现"的自然分工进化，还是被"制造"出来的职位产品？（参见下文激励分析。）

**Fail 3：第 224-226 行的 Agent 公式，如果逆向检验：** `Agent = Model + Harness`。如果我反过来问：如果删掉 Harness 层，Agent 还剩下什么？答案：前两次浪潮的工具。但如果再深问：**Harness 层本身是否引入了新的失败模式？** 文章暗示 Harness 层是纯增益的（第 269 行"不同的 Harness 产生截然不同的 Agent 能力"），但逆向思考要求你问：Harness 层引入的编排复杂度、门禁配置错误、Agent 协调失败——这些负面因素在哪里？

**Pass 1（值得保留）：第 566-574 行"何时不需要 Harness Engineering"。** 这是文章唯一一个逆向思考的痕迹——承认了场景边界。但从芒格的标准看，它只覆盖了"不需要"的场景，没覆盖"用了反而更糟"的场景。后者的说服力比前者大 10 倍。

## 二、激励审计（Incentive Analysis）

> **"Show me the incentive and I'll show you the outcome."**
>
> 不问"这篇文章说了什么"，问"这篇文章为谁的利益服务？"

### 激励结构

**第一个激励：Harness Engineer 是一个新角色定义。** 新角色创造新市场——书籍、课程、咨询、认证、招聘标签。这篇文章实质上是这个新角色的**价值宣言**（value proposition）。不是评估工具，是推销角色。

你要问自己：如果这篇文章诚实地告诉你"Harness Engineer 目前没有一个被行业认可的职位、没有标准的薪酬基准、没有成熟的培训体系"——你的判断会不会不同？这些信息文章都没说，因为它不符合"推销角色"的激励。

**第二个激励：OpenCode 的品牌植入。** 第 178 行的 YAML 示例使用 `examples/workflows/feature-pipeline.yaml` 路径。第 263-269 行的工具对比表中，ChatGPT "只能聊天"，而 OpenCode "工程化 Agent 工作流"。第 565 行预告"为什么选择 OpenCode"（下一篇）。

这不是阴谋——这是正常的商业行为。但作为一个读者，你需要知道这篇文章是在一个 **OpenCode 生态的书籍中**写的。它的激励机制决定了它不会公允地评价非 OpenCode 方案。例如：Cursor 的 .cursorrules 和自定义 Agent 模式，Claude Code 的 hook 系统——这些已经是某种形式的"Harness"，但文章只说它们属于前两代。

**第三个激励：Hashimoto 和 Chase 的名人效应。** 第 210 行用"Mitchell Hashimoto（HashiCorp 创始人）"作为引用——为什么加括号？因为括号里的信息（品牌名）比概念本身更能说服读者。Munger 的规则是：**用权威代替论证，本身就是一种认知偏误。** 我没有说 Hashimoto 和 Chase 是错的。我是说**文章引用他们的权威，而不是他们的论证**。Hashimoto 给出的是一个定义（第 212-214 行），不是一个论证。一个定义不能作为证据。

### 量化

| 利益相关方 | 从本文获得的利益 | 隐藏的代价（未提及） |
|-----------|----------------|-------------------|
| 书籍作者/框架推广者 | 概念普及、角色定义权 | 如果框架被证明不足，作者损失声誉 |
| OpenCode | 获得工程化叙事的合法性背书 | 用户期望被拔高，可能失望 |
| Hashimoto/Chase | 思想领导力地位巩固 | 概念被教科书化后失去演化弹性 |
| **读者（你）** | **获得的是一部完整的认知** | **可能的认知锁定——接受一个未经时间检验的框架** |

**结论：这篇文章的激励结构强烈偏向于"让读者接受框架"而不是"让读者评估框架"。** 这两者的区别是：前者提供完整的认知闭合，后者鼓励持续的怀疑。文章选择了前者。

## 三、Lollapalooza 效应检测

> **多种认知偏误同时发力、相互强化——比单个偏误危险 100 倍。**

以下是这篇文章同时激活的偏误：

### 偏误 1：权威偏误（Authority Bias）

第 210 行"Mitchell Hashimoto（HashiCorp 创始人）"、第 224 行"Harrison Chase（LangChain 创始人）"——两个创始人名字压阵。一个普通的定义，经过创始人的嘴说出来，变成了"权威定义"。但如果文章说的是"一个匿名开发者在 Reddit 上发了这个概念"——你还会以同样的信任度阅读吗？

### 偏误 2：社会认同（Social Proof）

第 7 行"AI 编程工具在短短五年内经历了三次浪潮"——"大家都在这么转"。第 577 行"LangChain 团队实验"——"一线团队在验证这个方向"。这些信号叠加在一起，让读者形成"所有聪明人都在做这个"的感觉。

### 偏误 3：叙事谬误（Narrative Fallacy）

三个浪潮（Prompt → Context → Harness）是一个太漂亮的线性故事。Munger 的规则：**如果历史被讲成了 neat story，它大概率是错的。** 真实的 AI 编程进化是混沌的：波浪式的、重叠的、甚至有些倒退的。2024 年的 Cursor/Claude Code 不仅仅是第二代——它们也在做 Harness 的事（.cursorrules、Agent mode、hook 系统）。2026 年的 OpenCode 也不是最终答案——更好的模型可能让部分 Harness 层变得多余。

### 偏误 4：可得性启发（Availability Heuristic）

第 109-121 行的 `rm -rf test /` 例子——一个戏剧性的、具体的安全事故。这个例子的冲击力会让读者高估它在真实世界的发生概率。文章用最坏的例子来证明对话模式的"不可接受"风险，然后用工程模式的漂亮流程图（第 135-159 行）来"解决"。**这是修辞操作，不是工程论证。**

### 偏误 5：过度乐观偏误（Over-Optimism）

第 427 行：三大原则是"衡量一切 AI 工程实践的标准"——这是无限制的、普遍的声明。没有任何领域在早期就拥有"衡量一切"的标准。熵的增加方向是反方向的。

### Lollapalooza 场景模拟

一个典型的读者路径：

> "Token 成本确实失控"（可得性：想起自己的痛苦经历）
> → "Hashimoto 也在说这个"（权威偏误：大佬认可）
> → "五个瓶颈，每个都很准"（确认偏误：验证了自己的不满）
> → "三浪潮，第三浪潮就是它"（叙事谬误：一个 neat 的解释）
> → "三大原则、五大能力"（过度乐观：问题有答案了）
> → **决策：买这本书，投入时间学习 Harness Engineering**

这个过程里，读者被一步步引导到"接受框架"的方向。**每一步单独看都合理，叠在一起就形成了 Lollapalooza。** 第 577 行的 52.8%→66.5% 数据——在 Karpathy 审阅中这是可靠性问题的证据，在 Lollapalooza 效应中它只是"第三浪胜利"的又一个信号。

### 对策

每个框架文章都应该自带一个**反偏误清单**——不是在结尾轻飘飘说一句"读者请自行判断"，而是具体指出"你可能正在被什么偏误影响"。这篇文章没有。

## 四、Too Hard 筐测试

> **芒格的分类：Yes（能做判断）、No（不做）、Too Hard（太复杂，放弃）。**
>
> **大部分聪明人的问题是：把 Too Hard 的东西强行放进了 Yes 筐。**

### 放入 Too Hard 筐的声明

**筐 1：第 427 行"三大原则是衡量一切 AI 工程实践的标准"**

Munger 会问：**你能用这套标准去衡量一个没有 AI 的工程实践吗？如果能，那它就不是 AI 特有的；如果不能，那它就不是"一切"。** 一个声称衡量"一切"的标准，如果连自身的适用范围都定义不清楚——Too Hard，放筐。

**筐 2：第 103-133 行的"安全失控是对话模式最危险的隐患" + 工程模式通过权限控制/审计日志/沙箱隔离"将安全防护系统化"**

Munger 会问：**你见过哪个系统化的安全方案在初期没有被攻破过？** 安全不是"加上三个机制就解决"的问题。权限控制有配置错误、审计日志有盲区、沙箱隔离有逃逸。文章把"有了这三个机制"等同于"系统化了"——这是在用清单代替体系。Too Hard，放筐。

**筐 3：第 222-226 行`Agent = Model + Harness` 公式作为"可操作的 Agent 定义"**

Munger 会问：**你对"公式"的定义是什么？** 数学公式可以预测行为——F = ma 告诉你给定质量和加速度，力是多少。这个公式能预测 Agent 的行为吗？不能。它是一个**分类框架**，不是公式。叫它公式，让读者以为它比实际更有预测力。把一个分类体系包装成公式——Too Hard，放筐。

**筐 4：第 429 行"同样的输入，经过同样的工作流，得到同样质量的输出"**

Munger 会问：**你用什么保证"同样质量"？** LLM 的随机性是内生的，不是通过配置就能消除的。你能固定 Temperature 和 Seed（第 451-455 行），但你不能固定用户的 prompt 表述方式、不能固定触发上下文的微小变化。**可复现性在统计学意义上可能是真的，在工程意义上是一个近似值。** 文章把近似值表述为确定性——Too Hard，放筐。

**筐 5：第 503-553 行的"可改进"原则——"从每次运行中学习，持续优化"**

Munger 会问：**你的改进是否收敛？** 文章画了一个漂亮的改进循环图（第 521-532 行），但没证明每次循环改进的幅度是否递减——也没有数据说明改进到什么程度会达到平台期。一个不知道什么时候停的"持续优化"不是工程，是信仰。Too Hard，放筐。

## 五、论证质量检查

> **Munger 的标准：一个论证的强度取决于它有多容易被证伪，而不是它有多漂亮。**

### 断言代替论证

**第 11-12 行**：单纯依赖对话式交互带来四个根本性问题——Token 成本失控、失忆问题、质量不可控、重复劳动。

**问题**：这些是"问题"还是"文章声称这是问题"？前三个问题（成本、失忆、质量）在对话模式中有缓解方案——有经验的用户会清理上下文、写系统 prompt、分段任务。文章把它们描述为对话模式的**本质缺陷**，但实际上它们是**使用方式**的缺陷——不是对话模式本身的问题。把使用方式问题包装成本质问题，是论证的核心脆弱点。

**第 133 行**：工程化模式"通过权限控制、审计日志、沙箱隔离等机制，将安全防护系统化"。

**问题**：这一句压在一个段落的末尾，看起来像是一个结论。但它没有论证为什么这三个机制加在一起就等于"系统化"。一个系统化的安全方案需要：威胁模型→控制措施→验证机制→持续监控→事件响应。文章只给出了中间一环（控制措施），就说整个系统化了。这是跳跃。

### 过度声称（Overclaiming）

| 行号 | 原文 | 真实状态 | 评估 |
|------|------|---------|------|
| 7 | "第三次浪潮重新定义了开发者与 AI 的关系" | 这是一个观察，不是事实 | **过度声称**。定义是否被重新定义取决于你有没有定义的权力。一个概念诞生 4 个月，还没有到"重新定义"的级别。 |
| 427 | "三个原则是衡量一切 AI 工程实践的标准" | 这是作者设立的宣言 | **过度声称**。"一切 AI 工程实践"——包括 Goolge 的内部 AI 工程实践？包括 Tesla 的 FSD？包括 OpenAI 自己的研究流程？文章没有任何证据证明这些原则已经被当前实践采纳。 |
| 212-214 | Hashimoto 的定义以"编程的未来不是写代码，而是驾驭 AI"开头 | 这是一个断言（claim），不是事实（fact） | **注意**：这不是 Hashimoto 的问题——一个创始人有权发表看法。这是文章**把这个看法当作定义**的问题。在学术语境中，定义需要共识；在商业语境中，定义需要市场验证。这篇文章把一个人的看法等同于行业共识。 |
| 577 | LangChain 实验将 Agent 准确率从 52.8% 提升至 66.5% | 这是文章**唯一**的量化证据 | **这个数字被过度利用**。在本文中，它同时扮演了三种角色：(1) Harness 有效性的证明；(2) 52.8%≈对话模式的失败率暗示；(3) 第三浪潮胜利的信号。一个 13.7% 的提升被赋予了它无法承受的论证重量。 |

### 缺失的反面证据（Darwin 协议检验）

Munger 的 Darwin 协议：**一个好的论证不仅仅是找支持证据，而是主动找反面证据。如果找不到有力的反对论据，可能是你搜索得不够努力。**

这篇文章的正面证据：

| 证据 | 类型 |
|------|------|
| Hashimoto 的定义 | 权威引用 |
| Chase 的公式 | 权威引用 + 一个实验 |
| 五个瓶颈 | 观察/归纳 |
| 五大能力 | 框架设计 |
| 三大原则 | 框架设计 |

这篇文章可以找但没找的反面证据：

1. **更好的 prompt 是否也能解决大部分问题？** 文章声称四个瓶颈是对话模式的"本质缺陷"。但一个有经验的 prompt engineer 可能会说：上下文管理可以用摘要技术缓解、质量不可控可以用迭代式 prompt 改进、重复劳动可以用 prompt 模板解决。这些反驳没有被纳入讨论。

2. **工具自身的累积改进是否让 Harness 层变得多余？** 2026 年的 Cursor/Cline 已经有 Workflow、Agent 模式和大量上下文管理能力。换句话说，工具也在内化 Harness 功能。文章的框架暗示"你需要一个外部的 Harness 工程体系"，但现实可能是"最好的 Harness 就是工具的下一步迭代"。

3. **Harness 层维护成本 vs 收益？** 文章给出了清晰的收益框架（三大原则），但完全没有讨论维护成本——Skill 的版本管理、Workflow 的调试、Agent 编排的复杂性。没有成本讨论的收益讨论是**选购目录**，不是工程评估。

## 七、一句话

**这篇文章构建了一个漂亮的框架，但它最大的问题不是它说错了什么——是它说对的那些东西也被包装得过于漂亮，以至于读者不容易看到那些没说的东西。** 用芒格的葡萄干与粪便法则说：如果一篇文章同时包含一个真实的洞察（作者发现了 AI 编程工程化的需求）和一个隐藏的激励（推广角色定义和特定工具），你不能只挑好的部分吃。你得问：**整块东西我可以信任多少？**

我的建议是：保留核心洞察——对话模式需要工程化约束。但把框架从"我们已经找到了答案"降级为"这是一个有价值的假设"——然后增加：

1. 一个诚实的反偏误清单（让读者自己检查 Lollapalooza）
2. 一个"我会在哪三种情况下放弃这个框架"的逆向思考片段
3. 一段关于为什么要讨论维护成本而不是只说收益的诚实对话

**诚实不是软弱的。它是框架能经得起时间检验的唯一方式。**
