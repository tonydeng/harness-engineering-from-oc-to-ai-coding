# 综合评审：failure-cases.md

## 评审概述

**文章定位**：`src/01-introduction/failure-cases.md` — 以三个 AI 编程失败案例（数据库清空、凭证泄露、源码误删）展示不加以约束的 AI 编码风险，引向 Harness Engineering 安全体系。

**参与视角**：
- **Karpathy 视角**（工程现实主义）—— 认为案例真实但框架有问题：多数失败是经典 DevOps 卫生问题戴着 AI 面具，文章把标准最佳实践包装成 AI-native 方案
- **芒格视角**（逆向思考 + 25 种误判心理学）—— 认为文章是在用三个精心挑选的恐怖故事推销预定结论，而非真正从失败中学习
- **技术负责人视角**（安全规范决策者）—— 认为这是全书最有实操价值的一章，但预防措施过度绑定 OpenCode 削弱了通用教育价值

**综合结论**：三个视角存在明显的分歧——Karpathy 和芒格从工程/逻辑角度对框架提出根本性质疑，而 Tech Lead 从实用性角度给予高度评价。共同认可的是案例的真实感和教育价值，共同批评的是预防措施过度绑定特定工具和缺少"事后应急"覆盖。

---

## 各视角发现汇总

### Karpathy 视角（工程现实主义）

**核心发现**：Well-intentioned, but the framing makes me uncomfortable — most failures are classic DevOps hygiene wearing an AI mask.

| 关键洞察 | 详细说明 |
|----------|---------|
| 案例一 — Database Flush | 真实问题不在 AI：开发者在生产环境手动执行命令。根本原因是缺少生产访问控制、只读副本和数据库备份——这些都是 AI 出现前就该解决的问题 |
| 案例二 — Context Injection | 技术上可行，但称为"Context Injection Attack"言过其实。这是数据泄露放大问题，根因和 2015 年一样：把密钥贴进共享上下文。Git Hook 是好建议，但也是前 AI 时代的实践 |
| 案例三 — Permission Misconfig | `trust_all` 模式是虚构的——没有主流工具真有此模式。`rm -rf src/` 场景过于夸张，需要同时假设 AI 和开发者都极端不称职 |
| 预防措施评估 | 环境隔离（好，标准，前 AI 时代即有）；危险模式匹配（对常见模式有效，假阳性率高）；审计日志（必需品）；敏感信息检测（对已知格式有效，新格式失败）；Pre-commit hook（真正有效，但被用了多年） |
| "Harness 拯救世界"框架 | 结论图暗示"对话模式"不安全而 Harness 模式安全——但多数预防措施是标准 CI/CD 实践，且过度约束会创造摩擦让开发者绕过系统 |

### 芒格视角（逆向思考 + 25 种误判心理学）

**核心发现**：表面上在用逆向思考，实际上是用三个恐怖故事推销预定结论。先有解决方案，再找三个失败来证明必要性。

| 关键洞察 | 详细说明 |
|----------|---------|
| 表面 Inversion，实际 Reverse-Engineering | 结构：有解决方案（Harness 权限体系/敏感检测）→ 找三个失败案例 → 每个案例的"预防措施"指向已有方案。这是确认偏误组合拳，不是逆向思考 |
| 可用性偏误 | 三个案例都有共同特征：损失明确、责任人清晰（小王/小李）、戏剧性高、解决方案直接。易于记忆和传播 |
| 被忽视的更致命失败模式 | 软错误（性能差两个数量级且持续一个月 > DB 被清空更贵）；依赖幻觉（不存在的 npm 包被攻击者注册）；业务逻辑错误（AI 不问"需求合理吗"）；自动化偏误（有安全系统→开发者不再审查→反而增加风险） |
| 预防措施逐个检查 | 环境隔离配置错了呢？模式匹配后知后觉——AI 换一种方式呢？审计日志只记录不阻止。Pre-commit hook 容易被 `--no-verify` 绕过。操作预览要求开发者能看懂 AI 决策——开发者原本就因为看不懂才让 AI 做的 |
| 文章隐瞒的三件事 | (1) Harness 不预防的失败类型比预防的多（20+ 种只选了 3 个匹配的）；(2) 三个案例都是"没脑子"级别的错误——稍有常识就能避免；(3) 安全措施本身引入新风险（复杂 YAML 增加攻击面、开发者图方便用 trust_all） |

### 技术负责人视角（安全规范决策者）

**核心发现**：最有实操价值的一章，但 OpenCode 绑定削弱了通用教育价值。

| 关键洞察 | 详细说明 |
|----------|---------|
| 场景说服力 | 案例一（Flush）和案例二（凭证泄露）极为真实——任何用过 AI 编程的团队都可能发生。案例三需调低极端程度——`trust_all` 和 `rm -rf` 过于夸张 |
| 团队培训价值 ⭐⭐⭐⭐ | 三个案例结构一致，适合 30 分钟安全分享。案例一做必讲内容、案例二做互动讨论、案例三做反面教材（需说明"正常情况不会这么极端"） |
| 预防措施可操作性 ⭐⭐⭐ | YAML 配置看起来清楚但缺实施路径：谁写配置？怎么推广到全团队？有免配置的降级方案吗？Git Hook 是唯一不依赖特定工具的措施，但可被 `--no-verify` 跳过 |
| 正面评价 | 案例一命令太真实——任何 Django 开发者都会后背一凉；安全 Checklist（6 个检查项）是全文最高密度价值输出；Mermaid 对比图适合培训 PPT；总结表三栏对照复习价值高 |
| **最大不足** | 所有预防措施绑定 OpenCode YAML 配置。如果团队不用 OpenCode，这些代码示例的可迁移性很低 |

## 问题与建议（合并表）

| # | 优先级 | 问题来源 | 问题描述 | 建议 |
|---|--------|---------|---------|------|
| 1 | **P0** | Karpathy + 芒格 | 文章框架本质是"先有方案后有案例"的确认偏误——把标准 DevOps 最佳实践包装成 AI-native 方案 | 增加诚实披露：多数预防措施是传统工程实践（环境隔离、Git Hook、审计日志），AI 只是加速了暴露已有问题 |
| 2 | **P0** | Tech Lead | 所有预防措施绑定 OpenCode YAML 配置，非 OpenCode 用户无法使用 | 每个措施先写一句工具无关的通用原则，再附 OpenCode 实现示例 |
| 3 | **P1** | Karpathy | 案例三 `trust_all` 模式是虚构的——现实中无此模式 | 调低极端程度，或改成批量删除旧模块时误删活跃模块，并说明虽略有夸张但过度授权导致的误删在社区多次被报告 |
| 4 | **P1** | 芒格 | 只覆盖了 20+ 种失败模式的 3 种，且都是最易防、最好写的 | 增加"本文未覆盖的重要失败模式"段落，承认局限性 |
| 5 | **P1** | Tech Lead | 只有事前预防，缺少事中应急和事后复盘 | 在总结部分增加事故响应三步骤：立即隔离 → 评估影响 → 恢复 |
| 6 | **P1** | 芒格 | 未披露安全措施本身的局限性和新引入的风险 | 增加权衡分析：YAML 配置增加攻击面、过度约束创造绕过动机、审计日志占用成本 |
| 7 | **P2** | Tech Lead | 缺误报率和运维成本的讨论 | 补充说明 `delete` 等关键词在 SQL 中频繁出现会误拦截，需要调整规则粒度 |
| 8 | **P2** | Karpathy | 结论图暗示"对话模式"不安全而 Harness 模式安全——过于简化 | 增加诚实声明：安全措施有可用性成本，工程挑战在于平衡安全与效率 |
| 9 | **P2** | Tech Lead | 危险命令拦截假阳性风险未提及 | 增加说明：实际应用中需注意关键词在正常操作中的高频出现 |
| 10 | **P2** | Tech Lead | 敏感信息检测规则缺少中文场景 | 增加 `password=`、`密钥` 等常见中文 Key 名及 `jdbc:` 连接串格式 |
| 11 | **P3** | 芒格 | 未提及自动化偏误（有安全系统→开发者放松警惕） | 补充"安全系统的二律背反"段落——如何避免安全措施本身成为风险源 |

### 最终建议

**核心矛盾**：Karpathy 和芒格质疑文章的根本框架（把 DevOps 最佳实践包装成 AI 安全方案），而 Tech Lead 认可其实用价值。两条线并不冲突——文章可以保持现有案例结构，但需要增加诚实披露。

**必须做的修改**：
1. 每个预防措施前加一句工具无关的通用原则
2. 增加"本文未覆盖的其他失败模式"段落
3. 补充事故应急响应流程（事前→事中→事后全覆盖）
4. 调整案例三的夸张程度

**保留的亮点**：案例一（Flush 生产库）的真实感、安全 Checklist（408 行）、Mermaid 对比图（384 行）——这些是全文最高价值输出，无需改动。
