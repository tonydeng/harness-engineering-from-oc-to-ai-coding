### 数据研究视角：事实核查

**研究方法**: 直接验证 OpenCode schema + 配置文件分析 + 文件系统审计。

**配置字段验证**:

从 OpenCode 官方 schema 解构的 Config 对象，**不存在于 schema 的字段**：
- ❌ `validation` — 不是合法顶层键
- ❌ `gates` — 不存在
- ❌ `yolo`（作为对象）— schema 中无此键
- ❌ `risk_classifier` — 不存在
- ❌ `auto_fix` — 不存在
- ❌ `validation_signing` — 不存在
- ❌ `config_protection` — 不存在
- ❌ `fix_loop_protection` — 不存在

**Permission 配置格式验证**:

| 声明格式 | 实际格式 | 判定 |
|---------|---------|------|
| `{"allow": ["read_file"], "ask": ["shell"]}` | `{"read": "allow", "bash": "ask"}` | ❌ 格式错误 |
| 分组式 allow/ask/deny 数组 | 以 tool 为键，值为 allow/ask/deny per-tool | ❌ 逆转录 |

**正确格式**:
```json
{
  "permission": {
    "read": "allow",
    "edit": "allow",
    "bash": "ask",
    "glob": "deny"
  }
}
```

### Karpathy 视角：工程现实主义评审

**核心结论**: 本文存在**严重的事实性失真**——大量描述的"验证护栏"功能在 OpenCode 原生实现中**根本不存在**。

**❌ 虚构功能清单**:

| 功能 | 文章中声称 | 实际情况 | 判定 |
|------|-----------|---------|------|
| **YOLO 风险分类器** | 三级风险分类（高/中/低）+ 规则引擎 + 机器学习训练 | OpenCode YOLO 仅为 `"yolo": true` 布尔开关 | ❌ 完全虚构 |
| **三级质量门禁** | 硬性/质量/量化门禁配置块 | OpenCode 权限系统只有 allow/ask/deny 三级策略 | ❌ 完全虚构 |
| **自动修复循环** | max_attempts 配置、lint 自动修复循环 | OpenCode 没有自动修复循环 | ❌ 完全虚构 |
| **验证结果签名** | RS256 签名、时间戳验证、会话绑定 | OpenCode 没有验证结果签名机制 | ❌ 完全虚构 |
| **yolo.training 配置** | learning_rate、min_samples、forget_after_days | 实际 `yolo` 是 `boolean`，不是对象 | ❌ 虚构 |
| **yolo.custom_rules** | 配置对象 | 从无此字段 | ❌ 虚构 |

**✅ 版本号正确**:
- OpenCode v1.15.x ✅（v1.15.6, v1.15.10-13）
- OpenCode v1.16.x ✅（v1.16.0）
- OMO v4.5.x ✅（v4.5.0, v4.5.12）
- OMO v4.7.x ✅（v4.7.4）

**根源问题**: 作者把 YOLO mode（一个简单的权限绕过开关）错误地扩展成了一个带有规则引擎、机器学习训练的复杂风险分类系统。

**预计修复工作量（原版本）**: 删除虚构功能约 600 行 + 替换为实际功能约 200 行 + 修正跨章节引用 = **总重写约 50-60%**。

**总体评价**: "概念框架好但工程实现完全失真"——建议重写为"最佳实践建议"而非"现有功能说明"。

### Munger 视角：批判性思维评审

**审计方法**: 逆向思考 + 误判分析 + 边界情况 + 简化原则 + 心理偏误识别。

**关键假设检验结果**:

| # | 假设 | 判断 | 理由 |
|---|------|------|------|
| 1 | 基于规则的模式匹配足以区分高低风险操作 | ❌ 结构性缺陷 | 规则系统有界，盲区必然存在 |
| 2 | 自动修复成功率数据（99%/85%/60%/90%/40%）可靠 | ❌ 伪精确 | 无来源、无方法学、无置信区间 |
| 3 | 验证结果签名（RS256+30 天轮换）能防止伪造 | ❌ 工程上不可行 | 密钥管理无 PKI 支持，自签名循环论证 |
| 4 | 量化门禁阈值（加载<3s, 包<500KB）普遍适用 | ❌ 无依据 | 数字似乎是取整出来的，非数据导出 |
| 5 | STRIDE 威胁建模适用于此场景 | ❌ 过度设计 | 套用了不匹配的大框架 |

**逆向风险分析**:

最有价值的逆向问题："如果验证系统不存在了，攻击者能否察觉到？"
回答：**不能**。攻击者可：
1. 用 `--no-verify` 或等效标志跳过门禁
2. 直接编辑源文件（跳过 Agent 抽象层）
3. 使用另一个 AI 编程工具
4. 手动运行 `git commit` 绕过 CI hook

**确认疲劳（Security Fatigue）**:
文章描述的漏报处理（用户反馈→调整规则）构成正反馈循环：边界模糊→提升风险等级→更多确认提示→更快确认疲劳→更大漏报概率。

**未覆盖的边界情况**:
- ❌ 零测试项目：覆盖率门禁（80%）在无测试项目中永远失败
- ❌ 大型单体仓库：批量操作阈值（5 文件）在 monorepo 中每次改动触发确认
- ❌ 命令不存在：`npm run build` 在 Python 项目中不存在
- ❌ 规则优先级冲突：多个匹配规则谁胜出？
- ❌ 0day 依赖漏洞：`npm audit` 只能检测已知漏洞
- ❌ Agent 崩溃：无回滚机制
- ❌ 并发冲突：多 Agent 同时修改同一文件
- ❌ 验证系统本身的错误处理：LSP 崩溃、`npm audit` 超时

**简化建议**:
- STRIDE 威胁建模（~270 行）→ 浓缩为 3-4 个威胁场景
- 三级风险分类 → 两级（阻断/放行）
- 增加验证系统的性能影响分析、代码被拒后处理、自身健壮性评估

### TechLead 视角：架构完整性评审

**核心发现**: 概念合理，实现虚构；16 个示例文件全部缺失；超过工程化。

**架构完整性分析**:

| 维度 | 裁决 | 说明 |
|------|------|------|
| 定位与分层 | ✅ 正确但不完整 | 准入/准出对偶关系正确，但缺少验证→反馈→约束的闭环 |
| 自动修复循环拓扑 | ⚠️ 有问题 | 每次重试调用 LLM，3 次尝试最坏 4 次 LLM 调用，成本未讨论 |
| 配置准确性 | ❌ 16 个文件缺失 | 所有 quality-gates/、validation/、audit-logs/ 引用均不存在 |
| 门禁动作枚举 | ⚠️ 不一致 | 定义了 block/warn 两种模式，但 metric gates 引入了未定义的 review |
| 配置保护循环依赖 | ⚠️ 软性安全 | `0644` 权限对同一用户下的 Agent 无约束力 |

**技术可行性问题**:
- `npm audit --audit-level=high` 作为硬性门禁 ❌ — 大项目几乎不可能达到零 high-severity 审计警告
- 模式匹配分类器可被字符串拼接、Base64 编码、脚本间接执行绕过
- 沙箱机制 6 次提及但无技术实现说明（Docker? Firecracker? gVisor?）

**STRIDE 严重性评级争议**:

| 威胁 | 文档评级 | 建议评级 | 理由 |
|-----|---------|---------|------|
| Spoofing | 中 | **高** | 单点故障，验证系统是最后一道防线 |
| Information Disclosure | 低 | **中** | 验证日志含源代码差异、文件路径等敏感信息 |
| DoS（修复循环） | 中 | **高** | Token 成本 = 直接经济损失 |

**总结评分（原版本）**:

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构设计合理性 | ⭐⭐⭐⭐ (4/5) | 准入/准出分层正确 |
| 质量门禁分级设计 | ⭐⭐⭐ (3/5) | `review` 动作未定义 |
| 风险分类逻辑 | ⭐⭐ (2/5) | 纯模式匹配几乎无效 |
| STRIDE 分析准确度 | ⭐⭐⭐ (3/5) | 2 项严重性误判 |
| 配置示例正确性 | ⭐⭐⭐ (3/5) | 语法正确但文件全缺失 |
| 技术可行性 | ⭐⭐ (2/5) | 沙箱未定义、配置保护循环依赖 |

**总体评价**: "更像一篇**架构设计白皮书**而非**工程实现指南**。"
