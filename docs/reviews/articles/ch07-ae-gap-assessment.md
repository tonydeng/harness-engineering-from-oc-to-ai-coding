## AE-01：Agent 配置演进

**定义**：案例是否展示了 Agent 配置（AGENTS.md / opencode.json / 路由规则 / Skill 绑定）随项目生命周期的演化过程？

| 案例 | 评分 | 覆盖内容 | 缺失内容 |
|------|:----:|----------|----------|
| real-world-01 | 2 | `/init` 生成 AGENTS.md、完整 opencode.json 配置、Skill 加载、Team Mode 配置 | 配置是静态的，未展示"随着项目演进修改配置"的过程 |
| real-world-02 | 1 | 配置了 security-research 团队的 5 人并行审计配置 | 配置是一次性的，未展示迭代演进 |
| case-security-audit | 1 | 配置了红蓝 team 中的 Agent 角色和工具链 | 配置固定，未展示安全基线配置的更新演进 |
| case-full-pipeline | 2 | 配置了 4 阶段交接门禁、模型路由、多 Agent 并行配置 | 配置是设计时决定的，未展示运行时调整 |

| case-skills-marketplace | 2 | Skill 版本管理、frontmatter 标准化、CI/CD 发布门禁 | 侧重于 Skill 本身而非 Agent 配置 |

**结论**：AE-01 总体覆盖良好（6 个案例平均 1.83）。case-multi-model 是最佳参考。无需追加内容。

## AE-03：循环工程/Agent 可靠性

**定义**：案例是否涉及 Generator-Evaluator 模式、反馈循环、验证门禁、停止条件、错误处理、退化检测？

### 逐案例分析

| 案例 | 评分 | 覆盖内容 | 缺失内容 |
|------|:----:|----------|----------|
| real-world-01 | 2 | 5 路并行审查、测试覆盖率门禁、修复循环（审查→修复→重测）、quality gate | 未涉及停止条件定义、退化检测、超时/预算限制 |
| real-world-02 | 2 | 分阶段 quality gate（每期交付有通过标准）、"三可"原则、灰度 Feature Flag | 未涉及 Generator-Evaluator 分离、退化检测 |
| case-security-audit | 2 | 红蓝闭环（发现→修复→复扫→清零）、CVSS 自动评分、误报三层过滤 | 未涉及 Agent 行为退化检测、停止条件 |
| case-full-pipeline | **3** | 每阶段都有交接门禁（4 道）、自动代码审查、CI/CD 集成验证、测试覆盖率门禁、PR 最终门禁 | Generator-Evaluator 模式未显式分离 |

**结论**：AE-03 总体覆盖良好（平均 2.0），尤其 case-full-pipeline 的交接门禁设计是循环工程的典范。无需追加内容。

## 修复方案

### 推荐：在 case-full-pipeline.md 追加上下文工程小节

**理由**：
1. case-full-pipeline 有 4 个显式的"交接点"（需求→设计、设计→开发、开发→测试、PR 创建前），这些是上下文工程的天然切入点
2. 现有门禁只检查"格式完整性"（JSON Schema 校验、必填字段），但不检查"上下文传递完整性"
3. 该案例在全书中对 AE-03 覆盖最好（3 分），但在 AE-02 上为 0，补充上下文工程将使该案例成为 AE-02 + AE-03 的双标杆

**插入位置**：在 §5.4（PR 最终门禁）之后、§6（效果指标）之前

**内容类型**：通用指导段落（非虚构案例内容），聚焦上下文工程在流水线各阶段的应用

**内容要点**：
- 上下文在流水线不同阶段间的传递策略：需求上下文→设计上下文→实现上下文→测试上下文
- 每个交接点的上下文压缩（保留决策理由而非全部对话历史）
- 渐进式披露原则（每个阶段只接收它需要的上下文）
- 模型切换时的上下文丢失预防

### 备选：在 real-world-01.md 追加

**理由**：作为第一个案例，涉及 AGENTS.md 的创建和 opencode.json 配置。但该案例的"配置"偏向静态创建而非动态传递，插入上下文工程段落不如 case-full-pipeline 自然。

**决定**：选择 case-full-pipeline.md 作为插入目标。
