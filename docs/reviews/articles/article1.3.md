## 一、评审概述

本文是全书的理论脊柱，提出四大支柱（安全、可观测、编排、成本）、三阶段演进模型和 Martin Fowler 五分类法。三位审阅者一致认为框架识别了一个真实的核心张力（AI 速度 > 人类验证速度），但都在不同程度上指出了同一个根本问题：**理论框架越完整越优雅，和 messy reality 之间的裂缝就越大**。

| 视角 | 核心关注 | 总体评价 |
|------|---------|---------|
| **Karpathy** | 三阶段时间线真实性、四支柱映射关系、可观测性过度乐观 | 框架合理但太干净；可观测性的当前状态被显著高估 |
| **Munger** | 框架实质绑定 OpenCode 却声称通用、声称"工程"但缺乏工程证据 | 最多 40% 是扎实的观察，60% 是未经检验的信仰 |
| **TechLead** | 决策支持不足、无实施优先级、信息缺口大 | 框架清晰但落地断层明显——理解了"是什么"但不知"怎么做" |

## 二、各视角发现汇总

### Karpathy 视角——工程现实主义

**核心框架**：March of Nines / Software X.0 / 工程现实主义

**主要发现**：

1. **三阶段演进模型——教学工具有效，历史描述不可靠**：
   - Prompt Engineering 从来没有"结束"过——2026 年仍是最重要的单一技能
   - Context Engineering 也没有"结束"——1M token 上下文利用率仍是开放问题
   - Harness Engineering（2025-2026）的 timeline 恰好落在这本书的主题上——这个巧合让人怀疑
   - Timeline 图 vs "能力叠加"的文字说明在打架

2. **四个支柱 vs 四个权衡的 1:1 映射太干净**：真实工程中每个 pillar 和每个 tension 之间都是多对多的关系。安全影响可观测性（审计日志粒度）、编排（安全策略决定 Agent 并行度）、成本（安全检查消耗 Token）。把复杂问题归纳成四个 1:1 映射——clean abstraction 离 reality 更远。

3. **可观测性被显著高估**：AI 可观测性在 2026 年仍是一个 open problem，远未到"透明盒"。LLM 的"中间思考过程"（chain-of-thought, reasoning tokens）在当前 observability 框架下几乎不可观测。文章描述的"三个层次"是目标，不是当前可用状态。乐观了 2-3 年。

4. **"缺一不可"忽略了实际优先级**：在数学意义上可能对，但在工程意义上四个支柱在不同阶段有完全不同的优先级。P0 是编排，P1 是安全，P2 是可观测，P3 是成本。

### Munger 视角——多元思维模型

**核心框架**：激励审计 / 逆向思考 / Lollapalooza / 能力圈审计 / Too Hard 筐

**主要发现**：

1. **框架实际绑定 OpenCode 却声称通用**：四大支柱、三阶段演进、Agent-Skill-Workflow 三层抽象都围绕着 OpenCode 的具体架构设计。文章把自己包装成通用理论，实际上是特定工具的方法论宣言。作者的 IP 绑定在 OpenCode 生态上——框架越复杂系统，书的独特性越强。

2. **"工程"这个词的修辞功能**：把"配置管理"称为"工程"不会让它变成工程。真正的工程有可衡量的标准、可证伪的假设、可复现的数字。这篇文章在"工程"的旗帜下实际提供的主要是分类法和叙事。分类法有价值，但都不是工程。

3. **逆向思考——框架在什么情况下会造成伤害？**
   - 失败模式 1：仪式 > 产出——团队花更多时间配 Workflow 而非交付软件
   - 失败模式 2：虚假的信心——编排审计监控的系统仍可能产出持续错误的代码
   - 失败模式 3：过早固化——2024 的最佳实践 2025 就过时
   - 失败模式 4：多 Agent 复杂性税——多 Agent 系统在学术界和工业界都被公认是极难的

4. **Lollapalooza 效应**：社会认同（SWE-bench 驱动行业叙事）+ 过度乐观（"编排+可观测=可控"是巨大跳跃）+ 近期偏误（3 年历史不足以构成"演进规律"）+ 确认偏误（框架设计者从自己的观察出发）+ 被剥夺超级反应（"不引入框架就落后"的暗示）。这篇文章既是这个 Lollapalooza 的产物，也是它的推动者。

5. **Too Hard 筐**：多 Agent 编排是 AI 编程的下一阶段（工程可行性仍是开放问题）、可观测性解决信任问题（LLM 推理过程仍是黑箱）、框架是通用的（直接映射 OpenCode 架构）、安全支柱=权限+审计+沙箱（真正安全工程远不止此）、成本是可控的（Token 定价权不在你手里）。

6. **能力圈审计**：文章最好地描述了一个真实的问题（AI 速度 > 验证速度）。但声称"工程化"缺乏工程证据（无数据支撑核心主张、无可衡量成功标准、无反例讨论）。把工具选择包装成通用理论。过度承诺了 AI 的可控性。

### TechLead 视角——技术负责人决策

**核心框架**：团队采用、落地指导、决策支持

**主要发现**：

1. **决策支持价值部分可用**：可以做的决策包括工具分类选择（五分类法）、认知对齐（四个支柱作为讨论框架）、阶段判断（三阶段帮助定位团队当前阶段）。无法做的决策包括采用决策（无 ROI 数据）、优先级排序（四个支柱都很重要但哪个先做？）、风险判断（失败模式完全没提）。

2. **四大支柱适用性有梯度**：编排（最实用，直接对应日常协作问题）、安全（最容易忽视但最必要，但缺"怎么落地"）、可观测（最有潜力，但过于泛化，没说明 Agent 可观测性和传统应用监控的区别）、成本（最容易被低估但最现实——但对 Tech Lead 来说成本管理就是在省钱）。

3. **三阶段演进可信度存疑**：2021→2023→2025 的时间线大体合理，但 Stage 3 的探索期终点在哪里？Stage 3 的半数工具本身就是竞品——它们的"编排"能力是同一回事吗？引用没有链接——不能验证的引用等于没有引用。时间线对企业级采用偏激进——多数企业可能全部在 Stage 1。

4. **信息缺口是一大弱点**：缺少的内容包括——没有具体数字（Token 消耗量级？）、没有团队配置参考（3 人 vs 30 人方案不同）、没有失败案例分析（缺一个会出什么事？）、没有迁移路径（从零开始按什么顺序引入？）。

5. **真实问题清单**：从哪开始？投入产出比多高？适用团队规模下限是多少？Harness Engineer 和普通开发者差距有多大？有没有反面教材？会不会增加团队认知负担？

### 一句话结论

**三位视角的共同结论：这是一个优秀的理论框架说明，但不是一个合格的"框架采用决策支持"文章。** Karpathy 指出三阶段时间线和四支柱映射过于干净、可观测性被高估；Munger 指出框架实质绑定 OpenCode 却声称通用、在"工程"旗帜下提供的主要是分类法和叙事；TechLead 指出从"理解框架"到"落到团队"之间的断层基本是空的——没回答 ROI、风险、资源投入这几个最核心的问题。如果能在某处给出一个"Harness Engineering 采用决策框架"（含团队规模判断、优先级推荐、实施周期参考、ROI 估算模型），整个理论体系的可用性会大幅提升。
