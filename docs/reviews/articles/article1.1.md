# 综合评审：`what-is-harness-engineer.md`

> **目标文件**：`src/01-introduction/what-is-harness-engineer.md`（585 行）
> **合并日期**：2026-06-16
> **来源视角**：Karpathy（工程现实主义）、Munger（多元思维模型）、TechLead（技术负责人决策）

---

## 一、评审概述

本文定义了"Harness Engineer（驾驭工程师）"这一概念——从对话式 AI 编程到工程化 AI 编程的角色转变。三位审阅者一致认为文章的核心框架（三大原则、五大能力、`Agent = Model + Harness` 公式）方向正确，但都在不同层面上指出了文章对"问题难度估计不足"的共性问题：

| 视角 | 核心关注 | 总体评价 |
|------|---------|---------|
| **Karpathy** | 工程可靠性、March of Nines、AI 的本质随机性 | 框架正确，但未定义可靠性边界，缺少对 AI 幽灵本质的承认 |
| **Munger** | 激励结构、认知偏误、论证质量 | 好故事不等于好论证——漂亮的框架让人放松警惕 |
| **TechLead** | 团队决策支持、ROI、导入路径、风险透明度 | 概念定义清晰，但不足以支撑"是否采用"的团队级决策 |

---

## 二、各视角发现汇总

### Karpathy 视角——工程现实主义

**核心框架**：Software X.0 / March of Nines / Iron Man Suit vs Robot / LLM = 召唤的幽灵

**主要发现**：

1. **可靠性边界未定义**：文章引用 52.8% → 66.5% 的准确率提升数据，但 66.5% 连"一个九"（90%）的门槛都没摸到。Marche of Nines 框架下，指数级的可靠性差距不应被掩盖。

2. **`Agent = Model + Harness` 公式遗漏 Human-in-the-Loop**：在当前模型可靠性水平下，没有人类兜底的二维公式是不完整的。建议补充第三维。

3. **Iron Man Suit vs Robot 混用**：文章从"Harness=盔甲"的语义出发，却在能力展示时滑向了全自动流水线叙事。没有区分增强模式（L1-L2）和自主模式（L3-L4）。

4. **LLM 的"幽灵本质"未被正视**：质量门禁本身就是 LLM 驱动的（`code-review: approved-by-review-agent`），用 LLM 审查 LLM 输出等于用梦检查另一个梦。可复现性也是统计学的，非数学的。

### Munger 视角——多元思维模型

**核心框架**：逆向思考 / 激励审计 / Lollapalooza / Too Hard 筐

**主要发现**：

1. **逆向思考（Inversion）**：文章从未考虑"Harness Engineering 可能是错的"。三次浪潮是辉格史观——把当前最流行的概念放在历史终点。Hashimoto 的定义被当作权威背书，但概念诞生 4 个月被写入教科书式框架——这是营销信号而非验证信号。

2. **激励审计**：这篇文章实质上是新角色（Harness Engineer）的价值宣言，不是评估工具。OpenCode 品牌植入（YAML 路径、工具对比表）和名人效应（Hashimoto、Chase）都指向"让读者接受框架"而非"让读者评估框架"。

3. **Lollapalooza 效应**：权威偏误 + 社会认同 + 叙事谬误 + 可得性启发 + 过度乐观偏误 五种偏误同时叠加。典型读者被一步步引导到"接受框架"的方向。

4. **Too Hard 筐**：五个声明被放入"太复杂"筐——"三大原则是衡量一切的标准"、安全通过三个机制就"系统化了"、`Agent = Model + Harness` 被称为"公式"、可复现性的确定性表述、"可改进"原则的收敛性未证明。

### TechLead 视角——技术负责人决策

**核心框架**：团队采用决策、ROI、风险、导入路径

**主要发现**：

1. **ROI 数据空白**：唯一的量化证据是 52.8% → 66.5%，但 66.5% 的准确率会劝退技术负责人。没有效率提升倍数、对比实验数据、学习周期估算。

2. **团队导入路径缺失**：全文没有一个字关于 rollout 策略、分阶段导入、团队培训或成功标准。技术负责人无法在 3 周内让 pilot team 跑起来。

3. **能力建设路径空白**：五大能力听起来像一整套新技能。没有习得难度评估、培养时间估算、角色过渡路径（谁可以兼任？是否需要专职？）。

4. **没有风险透明度**：缺少"负面清单"——过度工程化、知识沉淀的"铁锈"、Agent 编排的"专家盲区"、质量门禁的"假安全感"、团队分化。

5. **与现有工程体系衔接缺失**：没有说明如何与 Code Review、CI/CD、Sprint 流程集成。

6. **细节问题**：第 536-551 行的 Metrics 明显是编的数字；概念仅 4 个月但未标注为早期采纳决策；YAML 示例缺少错误处理说明。

### 一句话结论

**三位视角的共同结论：文章的核心框架（Agent = Model + Harness、三大原则、五大能力）是站得住的，但它在"说服读者相信这个框架"上花的力气远多于"帮助读者理解这个框架的真实边界"。** Karpathy 指出没说明可靠性天花板，Munger 指出没给读者反偏误工具，TechLead 指出没回答"我该怎么落地"——这三个缺口指向同一个问题：文章需要从"概念宣言"升级为"决策支持文档"。
