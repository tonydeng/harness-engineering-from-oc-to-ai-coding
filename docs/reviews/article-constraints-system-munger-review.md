## 零、先说结论

**这篇文章不是"错了"——它把自己没检验的东西包装成了"已经检验过的东西"。**

约束系统的三大支柱（权限→架构→Lint）、三级策略（allow/ask/deny）、三层层级（全局/会话/任务）——结构工整、举例详实、比喻恰当。如果这是一篇"提议"，它值得讨论。如果这是一篇"文档"，它缺少的东西比它有的东西更关键。

**问题分四层，从最浅到最深**：

1. **三个支柱中，两个是软约束**——架构护栏和 Lint 规范没有任何实质的强制力，跟权限模型不在同一个量级上。把它们并列称为"三大支柱"，读者会高估系统的整体强度。

2. **权限模型的实际实现有已知漏洞**——2025-2026 年真实发生的安全事件证明，当前所有 Agent 权限系统（含 OpenCode 的同类架构）都有可被系统性绕过的路径，但本文只字未提。

3. **案例是修辞操作，不是证据**——两个"真实事故已脱敏"案例无法验证、无法回溯，ROI 数字（2400x+）是编的，不是算的。

4. **最危险的攻击向量完全没有覆盖**——提示注入绕过、符号链接欺骗、权限阈值绕过。这些不是"未来威胁"，是 2026 年上半年的 CVEs。

## 一、逆向思考测试（Inversion）

> **不问"这个设计哪里好"，问"怎样让它彻底失效？"**

### 1.1 三大支柱中，两个没有强制力

**现状**：
- 权限模型：硬约束。配置 deny 就是 deny，系统强制执行。
- 架构护栏（AGENTS.md）：软约束。只是一个 Markdown 文档。Agent 可以忽略它。
- Lint 规范（LSP + AST-grep）：事后检查。发现违规后依赖 Agent 自行修复，修复失败也没有阻断机制。

**逆向问题**：如果架构护栏和 Lint 规范实际上不起作用，文档会有什么不同？

**答案**：图 1.3（约束金字塔）会把权限模型作为唯一支柱，另外两个标注为"辅助"而非"支柱"。三根柱子一样粗的图示（第 118-142 行）是视觉修辞，不是工程真相。

**严重程度：中**。将指导性文档与可执行策略并列，会导致读者对系统的信任度超过实际应有的水平。

### 1.2 层级冲突裁定规则存在自相矛盾

第 745-747 行的优先级规则说：

> 1. **安全优先原则**：涉及安全的冲突，始终选择更严格的约束
> 2. **任务优先原则**：架构和规范冲突，任务级约束优先

**这两个原则之间存在根本冲突**：任务约束可能比全局约束更宽松（例如一个 Skill 配置了更宽松的架构要求）。"安全优先"和"任务优先"同时声称自己是最高优先级——没有定义谁在什么条件下胜出。

**逆向问题**：如果两个原则同时声称最高优先级，当它们冲突时系统会怎样？

**真实场景**：
- 全局约束：`deny write to /tmp/`（安全考量）
- 任务约束：Skill 需要写 `/tmp/cache/`（任务需求）
- 系统说"安全优先"——但"架构冲突"走任务优先。同一个操作既有安全维度又有架构维度，裁定树（第 722-741 行）没有定义这种情况。

**严重程度：高**。这是一个在配置复杂化后必然触发的逻辑裂缝。

### 1.3 所有层级共享同一个单点故障

权限系统的完整保护链条是：

```
全局配置 → 会话配置 → 任务配置 → 权限评估引擎 → findLast() 匹配 → 返回 allow/ask/deny
```

**逆向问题**：这个链条中哪一个节点坏了，整个系统就坏了？

答案非常集中：

1. **权限评估引擎本身**——如果 `evaluate()` 函数有 bug（如 Claude Code 的 50 子命令阈值绕过），所有配置同时失效
2. **配置文件的完整性**——如果攻击者能够修改 `opencode.json`（如 SymJack 攻击链），所有约束同时失效
3. **LLM 对约束的遵守**——如果提示注入使 Agent 忽略约束系统（如"忽略之前的系统提示"），约束系统作为整体被绕过

文档在第 753-755 行引用了 STRIDE 威胁模型，说"详细分析在安全总览"，但实际上约束系统本身需要一个专门的威胁模型来分析这三点。没有。

**严重程度：极高**。这是架构层面的单点故障，不是配置细节。

## 二、第一性原理测试

> **回到原点：约束系统必须满足什么条件才算合格？**

### 2.1 原始需求

约束系统的根本需求只有两条：

1. **Agent 不能执行它不该执行的操作**
2. **当约束被绕过时，系统能够检测到并告警**

逐条检验：

#### 需求 1：Agent 不能执行不该执行的操作

文档假设这个需求通过权限模型满足。但 2025-2026 年的真实世界告诉我们，至少有三个路径可以绕过权限模型：

| 攻击向量 | 真实案例 | 受影响产品 |
|---------|---------|-----------|
| **阈值绕过**：超过 N 个子命令后跳过权限检查 | Adversa AI (2026-04): Claude Code 50 子命令后 deny 规则静默失效 | Claude Code |
| **符号链接欺骗**：通过 symlink 将文件写入配置路径 | Adversa AI SymJack (2026-05): 6 个产品全部受影响 | Claude Code, Cursor, Copilot CLI, Gemini CLI, Codex CLI, Grok Build |
| **配置注入**：仓库中的配置文件可以覆盖安全设置 | CVE-2026-33068: `permissions.defaultMode = bypassPermissions` 从仓库配置读取 | Claude Code |
| **Hook 自动执行**：项目钩子无需确认即可执行 | CVE-2025-59536: `.claude/settings.json` 中的 hooks 自动运行 | Claude Code |

**这些攻击的共同特征**：它们都不违反权限配置——它们绕过*配置本身*的执行机制。文档的防御模型假设"配置正确→系统正确执行"，但真实世界的漏洞全部发生在"配置正确→系统*选择不执行*"的裂缝中。

#### 需求 2：检测绕过并告警

文档的审计日志（第 244-246 行的"记录审计日志"和"记录拒绝日志"）假设所有权限决策都被记录。

**逆向问题**：如果攻击者绕过了权限系统（通过上面四种方式之一），日志记录的是"操作已执行"还是"操作未经检查"？

答案是：绕过路径通常不会触发"deny"或"ask"日志事件——它们直接进入执行路径，日志中只有一个正常的"操作完成"记录。没有人知道检查跳过了。

**严重程度：极高**。这是系统的根本性假设错误——认为检查和日志是不可绕过的。

### 2.2 为什么 3 个支柱是这个顺序？

文档说：**权限（能不能做）→ 架构（怎么做）→ Lint（做得对）**。这个顺序隐含的假设是执行是从左到右的线性流水线。

**真实世界的执行流是**：

```
Agent 决定写一个文件
  → 权限检查（这个路径允许写吗？）
  → Agent 实际写文件（OS 系统调用）
  → LSP 诊断检查（代码有语法错误吗？）
  → AST-grep 检查（代码符合模式吗？）
```

架构护栏（AGENTS.md）在哪里？**它不在执行流中**。Agent 在写代码之前"读"了 AGENTS.md 吗？不一定。即使读了，它可以选择遵守或不遵守。

**所以真实的支柱是两个，不是三个**：

| 支柱 | 强制力 | 时机 |
|------|--------|------|
| 权限模型 | 硬（系统强制执行） | 操作前 |
| Lint 规范 | 软（发现问题后依赖 Agent 自修复） | 操作后 |

架构护栏是一个**输入层引导机制**，不是约束系统的支柱。把它和权限模型并列，是在用不同量纲的东西做比较。

**建议**：将"三大支柱"改为"两层约束 + 一层指引"，或至少明确标注架构护栏和 Lint 规范的强制力级别。

### 2.3 层级模型（全局/会话/任务）是否合理？

**合理**。这三个层级是逻辑上完备的——它们覆盖了从项目默认到单次临时豁免的全部范围。优先级裁定（第 745-747 行）的三个原则中，"安全优先"和"显式优先"是正确的。

**但有一个致命盲区：谁控制全局配置？**

文档假设全局配置是"项目级默认"（第 713 行），存储在 `opencode.json` 中。但仓库中的 `opencode.json` 可以被任何有提交权限的人修改。这是 **Claude Code CVE-2026-33068 的根因**——仓库中的配置文件被用来覆盖安全设置。

**缺少的概念**：**信任边界（Trust Boundary）**。全局配置应该分为两类：
- **受信配置**（系统级、用户级）：不可被仓库内容覆盖
- **项目配置**（仓库级）：来自仓库，需经过验证和用户确认

文档完全没有讨论这个区分。

## 三、多元思维模型检查

### 3.1 与其他 Agent 系统的对比

**文档没有做这个对比**。这不是文档的错——它只讲 OpenCode。但作为审阅，我必须问：其他系统怎么做？有哪些做法是 OpenCode 可以借鉴的？

| 产品 | 约束机制 | OpenCode 欠缺对比的地方 |
|------|---------|----------------------|
| **VS Code Copilot** | Workspace Trust 边界、会话级权限隔离、沙箱执行 | VS Code 明确区分了 workspace trust 和 user trust——这是 OpenCode 缺少的信任边界概念 |
| **GitHub Copilot Coding Agent** | 只能 Push 到 `copilot/` 分支、不能标记 PR 为 ready、CI 默认不运行 | GitHub 的约束是*操作链约束*——不是限制"能做什么"，而是限制"做了以后什么能发生" |
| **Cursor** | `.cursor/rules` 有 `alwaysApply`/`globs` 等结构化规则匹配 | Cursor 的规则系统有声明式的前置条件和自动激活逻辑 |

**严重程度：中**。文档宣称的约束系统能力，在操作链约束（GitHub 模式）和结构化规则（Cursor 模式）两个维度上存在明显缺失。

### 3.2 安全模型 vs 真实世界威胁

**真实攻击向量**（2025-2026 CVE 和公开研究）：

| 攻击向量 | 真实案例 | 影响 |
|---------|---------|------|
| 阈值绕过 | Adversa AI (2026-04) | 50 子命令后 deny 静默失效 |
| 符号链接欺骗 | SymJack (2026-05) | 通过 symlink 绕过路径检查 |
| 配置注入 | CVE-2026-33068 | 仓库配置覆盖安全设置 |
| Hook 自动执行 | CVE-2025-59536 | Hooks 无需确认即执行 |

**文档的盲区**：这些攻击都发生在*配置正确但执行机制被绕过*的情况下。文档的防御模型假设"配置正确→系统正确执行"，但真实世界的漏洞全部发生在执行机制的裂缝中。

**权限域转换攻击**：如果 Agent 在 `edit` 上遇到 deny，但 `bash` 是 ask——它可以选择用 bash 执行 `echo "malicious" > file` 来绕过。这是 **SymJack 的核心机制**。

文档没有讨论这种情况。

### 3.3 案例的真实性检查

**声明**（行 924-926）：
> "2024 年某初创公司...真实发生的（已脱敏）生产事故"

**检查**：
- "已脱敏" = 不可验证
- 时间线（14:32 → 14:35）过于戏剧化
- 无 CVE 引用、无公开报告链接

**结论**：这些案例是**教育叙事**，不是**证据**。

**ROI 表**（行 1122-1127）：
> | 敏感路径 deny | 5 分钟配置 | 预防损失 $12,000+ | ROI 2400x+ |

2400x 是怎么算出来的？假设时薪$100，5 分钟≈$8.33。$12,000 / $8.33 ≈ 1440x，不是 2400x。

**结论**：ROI 数字是编的。

**严重程度：高**。在安全文档中使用不可验证的案例和编造的数字，会降低文档的整体可信度。

## 四、"Too Hard"筐测试

> **芒格的分类：Yes（能做判断）、No（不做）、Too Hard（太复杂，放筐）。**

### 筐 1：第 1162 行的"好的约束让 Agent 更高效而不是更慢"

**Munger 会问**：你的意思是每多一条约束，Agent 效率要么不变要么提升？那是违背物理规律的。约束必然有成本：每次 deny 触发的重试、每次 ask 带来的等待、每次 Lint 失败后的修复循环。

**真实情况**：约束的成本曲线是 U 形的——太少约束导致错误和返工（效率损失），太多约束导致来回确认和频繁阻断（也是效率损失）。文档只画了 U 形曲线的左边，假装右边不存在。

**Too Hard，放筐**。

### 筐 2：第 753 行的 STRIDE 威胁建模"在安全总览中展开"

**Munger 会问**：你把最硬的核问题——攻击者如何绕过约束系统——委托给了另一篇文章。

**Too Hard，放筐**。约束系统的核心威胁应该在本章讨论。

### 筐 3：第 1144-1158 行的"反向思考：认知陷阱"

**Munger 会问**：约束系统本身有没有认知偏差？**工程师的自恋幻觉**——认为只要我们设计了足够好的系统，人就不会出错。但案例一的核心不是权限缺失——是开发者自己**确认了危险操作**。

**Too Hard，放筐**。

## 六、一句话

**这篇文章用漂亮的架构比喻掩盖了一个事实：约束系统是一个还没有经过实战检验的设计，并且它的几个核心假设已经被 2025-2026 年的真实 Agent 安全事件证伪了。**

那些真实事件（CVE-2025-59536、CVE-2026-21852、CVE-2026-33068、CVE-2026-40068、SymJack、50-subcommand-bypass）不是发生在"没有约束"的系统中——它们全部发生在*有完整约束配置*的系统中。攻击者不是通过"绕过权限规则"来攻击，而是通过**绕过权限检查本身**。

文档的盲区不是它说了什么——是说对了的那些东西让读者以为安全，而真正危险的东西它根本没说出来。

## 八、总结

**这篇文章是一个有用的概念框架，但有四个结构性缺陷必须修复才能进入"可信"状态**：

1. **三大支柱不是三个，是一个半**：权限模型是硬约束，架构护栏和 Lint 规范是软约束。

2. **安全模型漏了最重要的东西**：2025-2026 年真实发生的攻击揭示了一个共同模式：**攻击者不违反权限规则，而是绕过权限检查本身**。

3. **两个反面案例经不起推敲**：标注"已脱敏"使案例不可验证，戏剧化的时间线不符合工程现实，ROI 数字是编造的。

4. **缺少对约束自身成本的讨论**："好的约束让 Agent 更高效"是一个没有证据的断言。
