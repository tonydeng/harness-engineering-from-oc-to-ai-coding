# Karpathy 式审阅：`why-opencode.md` — 工程现实主义视角

> 以 Andrej Karpathy 的 engineering realism 框架审阅本文。
> 审阅者：Karpathy 思维模型（基于公开言论推断，非本人）

---

先说结论：这篇文章是一篇诚实的营销文章。它比大多数竞品对比文章做得好的地方是——它确实写了局限性和选型建议速查表（5.2节），不是纯吹捧。但"诚实营销"和"工程真相"之间有一个根本差距，这个差距用 march of nines 框架一下就看出来了。

---

## 1. March of Nines：对比表的 ✅/❌ 是工程判断还是营销话术？

这个问题本身就很有意思。这个对比表（1.2节，13 款工具）看起来非常系统——12个维度、13个工具、156个格子。作为一个重度表格用户我第一反应是 "wow, that's thorough"。但仔细看下去，问题就露出来了。

**问题一：OpenCode 的 156 个格子里几乎没有 ❌。**

OpenCode 在 12 个维度里拿了多少个 ❌？零。全部是 ✅ 或 ⚠️。而它所有的竞品——至少在每个维度都至少有一个 ❌。这意味着什么？要么 OpenCode 在所有维度上都优于或等于所有竞品（那它应该统治市场，但它没有），要么这张表的设计本身就有 bias。

具体案例：

- **Provider 自由度**：OpenCode ✅（75+ Provider），Cursor ✅（多模型 自由切换）。但 Cursor 的 "多模型" 支持 OpenAI/Claude/Gemini 等多个主流模型，对 99% 的用户来说已经够用了。75+ 里大部分是 obscure providers，这不是一个线性的 "more = better" 函数。把两者都标成 ✅ 没问题，但暗示 OpenCode 在这里有本质优势是 overclaim。

- **可观测性**：OpenCode ✅（完整日志 Token 追踪），Cursor ❌（黑盒 无日志），Windsurf ❌（黑盒 无日志）。这个我直接质疑。OpenCode 作为一个 CLI 工具确实有详细的 stdout logging，但 "可观测性" 作为一个工程维度——包含 tracing、metrics、structured logging、correlation IDs——这些事情 OpenCode 支持多少？把 "terminal prints things" 标成 ✅ 而竞品标 ❌ 有不诚实之嫌。

- **安全架构**：OpenCode ✅（沙箱隔离 权限控制）。OpenCode 的 sandbox 具体怎么工作的？代码是在子进程运行吗？有没有 container 隔离？权限控制到什么粒度？文章没有说。而 "无沙箱" 作为一个批评竞品的理由，60% of users don't actually need proper sandboxing——这是一个存在性声明，不是一个实用性评估。

**问题二：✅/⚠️/❌ 是分类不是评测。**

这个评级系统的问题是它只告诉你某个能力 "存在还是不存在"，不告诉你 "好到什么程度"。一个有一个简陋 plugin API 的工具和一个有 20+ Hook 点 + MCP + Skills 的工具都标 ⚠️ "有限扩展"——但这是两个不同的世界。同样，标 ✅ 的工具之间也有巨大的质量差异。

**March of Nines 的核心问题**：这些评级没有一个涉及可靠性的尾部行为。即便 OpenCode 真有 20+ Hook 点——它们在生产环境下的稳定性如何？边缘情况下（并发调用、大文件处理、网络中断）表现如何？这些问题不回答，这些 ✅ 的意义就只是 "feature exists" 而不是 "feature works reliably at scale"。

## 2. Iron Man Suit vs Iron Man Robot：这篇文章站在哪一边？

坦白说——它在两边都站了，但没意识到自己在两边都站了。

**Iron Man Suit 的一面**：
- 文章强调 OpenCode 是 "工程化平台"、"驾驭工程师" 的工具，人类开发者是主导者
- 提到 "人类在关键决策点的控制权"——虽然只是间接暗示
- OMO 架构设计保留了人类的审查角色

**Iron Man Robot 的一面**：
- Agent 被定义为 "执行器"：理解任务意图后自主执行多步操作（line 273）。"自主执行" 这个词——在这个语境里——更接近机器人而非套装。
- Team Mode 的 sequence diagram（line 565-590）：用户只在 "提交任务" 和 "任务完成" 两个节点出现。整个中间过程——任务分解、代码实现、安全审查、测试生成——全部是 Agent 自主完成。这不是 Iron Man Suit，这是 "set it and forget it"。
- 文章没有讨论：当 Build Agent 生成了有问题的代码、Plan Agent 做了错误的规划、多个 Agent 给出了冲突的输出时——人类如何介入？介入成本多高？

**Imo，这篇文章最缺的一个讨论是**：OpenCode 的 Agent 自主度目前处于什么水平？它在 march of nines 的哪个位置？它会在 90% 的场景下正常工作，然后 10% 的场景下出现完全意料之外的错误（锯齿状智能）——这是 Agent 产品的普遍现状。文章没提这件事，这是个问题。

## 3. Software X.0："Copilot vs Agent" 二分法准确吗？

```
|             | Copilot（补全器）          | Agent（执行器）            |
| 交互方式    | 被动响应：基于光标位置建议  | 主动执行：理解任务意图后操作 |
| 操作范围    | 单文件、单位置              | 多文件、多步骤、跨工具      |
| 上下文理解  | 局部上下文（当前文件）      | 全局上下文（整个项目）      |
| 任务复杂度  | 简单补全、函数生成          | 复杂重构、跨模块修改        |
| 可控性      | 低：接受或拒绝              | 高：可设计工作流、设检查点  |
```

（lines 270-277）

这个表的问题不是它错了——是说它把连续谱抽象成了二分法，而且这个二分法对自己有利。

**问题一：Copilot 也在进化。** GitHub Copilot 从 2021 年的纯补全器演进到了今天有 Copilot Chat、Copilot Workspace、agent mode 的产品。分类学上把它钉在 "补全器" 的柱子上是不准确的。2026 年的 Copilot 已经不是一个纯补全器了。

**问题二："可控性 高" 这个判断可疑。** OpenCode 的 Agent 确实支持 workflow 设计和检查点，但 "可控性" 的意思是——当你需要对 Agent 的输出做细粒度修改时，你的工具和工作流是否支持你快速定位和修改。用 Agent 系统做复杂任务时，调试 multi-agent 工作流比调试一个 copilot 的建议要难得多。说 Agent "可控性 高" 而 Copilot "可控性 低"——这个在工程实践上值得商榷。

**问题三：二分法忽略了中间态。** Claude Code 是 "自主执行 单 Agent"，Cursor 是 "对话式 单 Agent"——但 Cursor 也有 agent 能力，Claude Code 也做补全。工具能力的边界是模糊的，强行分类会误导读者的预期。

从 Software X.0 的角度看，这个二分法更像 Software 2.0 的思维（给系统贴上分类标签）而非 Software 3.0 的思维（理解工具的本质是对语言的不同层级的抽象，而不是功能的二分）。

## 4. 工程现实主义：局限性够诚实吗？（4.1-4.4）

四个局限性写得比我预期的好。它们没有被藏在某个角落——独立成节，标题叫 "诚实告知"，占了大约 100 行。对于一个开源书籍的宣传文章来说，这已经比 99% 的同类文章诚实了。

**写得好的部分**：
- 4.1 终端体验问题——事实清楚，而且给出了缓解方案
- 4.2 学习曲线——给出了具体的时间估算（1-2h / 1-2d / 1-2w / 1-2m），这是可量化的
- 4.4 其他局限——Windows 支持、大型项目性能、小众语言支持——这些是真实的痛点
- 5.2 选型建议速查表在特定场景（前端开发者、性能追求）下推荐竞品，这个值得尊重

**缺失的部分（这是关键）：**

**4.1 缺失：Agent 可靠性的尾部行为。**
文章讨论了 "终端界面体验" 和 "学习曲线"——这些都是显性问题。但隐性问题——Agent 在复杂任务上的实际成功率、系统性失败模式、多 Agent 协作的冲突处理成本——这些只字未提。March of nines 告诉我：真正让你痛苦的永远是 1% 的情况。如果 OpenCode 的 Build Agent 在 95% 的情况下工作良好，但有 5% 的情况会产生完全不可预期的输出——对于一个生产系统来说，那 5% 可能才是你真正需要花时间去处理的。文章对此保持沉默。

**4.2 缺失："Provider 自由" 的真实成本。**
文章宣传 "75+ Provider 自由切换" 是一个优势（2.2节），但切换 Provider 的隐性成本是什么？不同模型的 prompt 格式、system prompt 灵敏度、输出风格完全不同。把同一个 workflow 从 Claude 换到 DeepSeek，几乎一定需要调整 prompt。这不是一个 "零成本切换" 的场景。文章没有谈论这个。

**4.3 缺失：「50-70% 成本降低」（line 231, 457）的方法论。**
这个数字出现了两次，但没有 citation、没有 methodology、没有假设说明。是做 benchmark 得出来的？还是拍脑袋？在这个行业里，没有方法论的数字就是 marketing。如果在做对比评测时写这种数字，应该提供一个计算示例。

**4.4 缺失：自托管的运维成本。**
文章宣传 "可自行托管：企业可在内网部署"（line 219）作为一个优势，但没有讨论需要什么样的基础设施、运维能力和人力投入来维持一个自托管的 OpenCode 集群。对于选型决策来说，这部分成本可能是决定性因素，但被完全忽略了。

**4.5 缺失：社区风险。**
"160K+ GitHub Stars" 被列为优势——但开源项目的风险恰恰是 "community-driven" 意味着没有 SLA、没有商业支持、breaking changes 可能随时发生。文章站在 2026 年 6 月的时间点宣传社区的活跃度，但在做技术选型时，"如果核心维护者离开" 或 "项目方向突然改变" 是真切的风险。这件事应该至少被提到。

---

## 总结：这篇文章值不值得读？

值得。对于一个想了解 OpenCode 的读者来说，这篇文章覆盖了四个核心优势、给出了工具生态的对比框架、诚实地列出了部分局限性，而且推荐了竞品适用的场景。这是很好的入门文章。

但从工程现实主义的立场看，它的核心问题是对 OpenCode 的局限性做了不对称的坦诚——坦率地讨论了界面体验和学习曲线这些相对表面的问题，但回避了 Agent 系统的实际可靠性、多模型切换的隐性成本、自托管的运维负担这些更深层的工程问题。

对比表中的 ✅/❌ 判断体系在营销语境下是可以接受的——每家产品都会美化自己的定位。但如果你用 march of nines 的标准来看，这些判断衡量的是 "功能是否存在"，而非 "功能在真实工程场景下是否可靠"——而后者才是工程项目选型的真正依据。

就这样。
