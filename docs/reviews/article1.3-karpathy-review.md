# Karpathy 式审阅：`harness-engineering-theory.md` — 工程现实主义视角

> 以 Andrej Karpathy 的 engineering realism 框架审阅本文。
> 审阅者：Karpathy 思维模型（基于公开言论推断，非本人）

---

先说结论：这篇文章是全书的理论脊柱。它做对了很多事——四个支柱的框架是合理的，AI 编程的核心张力是真实的，Martin Fowler 的五分类法是实用的。对于一本要建立系统性论述的书来说，这些骨架搭得不错。

但理论框架有一个常见陷阱：**框架本身越完整、越优雅，和 messy reality 之间的裂缝就越大。** 这篇文章的裂缝在哪里？下面用我的几个框架来拆。

---

## 1. 三阶段演进：叙事很漂亮，真实性需要检查

文章提出一个三阶段模型：Prompt Engineering (2021-2023) → Context Engineering (2023-2025) → Harness Engineering (2025-2026)。

让我从几个角度看看这个模型。

**阶段一：Prompt Engineering (2021-2023)**

This is fine. 2021 年 Copilot 发布，prompt engineering 成为显学。2023 年 GPT-4 发布，few-shot、chain-of-thought、role-playing 这些技术成熟起来。时间线合理。

**但**——prompt engineering 从来没有 "结束" 过。我 2026 年还在写 system prompt，还在调温度参数，还在想 3e-4 是不是最佳学习率的事（好吧这不一样，但你们懂我意思）。做 AI 产品的人都知道，prompt engineering 仍然是让模型输出靠谱的最重要的单一技能。把它标记为一个 "已结束的阶段" 有误导性。

**阶段二：Context Engineering (2023-2025)**

这个命名很好。我同意把 context 管理看作一个独立工程领域的价值。Claude 2 的 100K 上下文在 2023 年确实改变了游戏规则，2024-2025 年 RAG、long-context、multi-file editing 成了标配。但这个阶段也没有 "结束"——实际上 context engineering 是当前最重要的 open problem 之一。1M token 上下文的利用率如何？retrieval 的效果曲线是怎样的？这些问题远没有被解决。

**阶段三：Harness Engineering (2025-2026)**

这里我最有意见，所以多说几句。

> "2025 年，Claude Code、OpenCode + OMO 等工具的出现标志着 AI 编程进入 '驾驭工程' 探索期。"

这个声明的 timing 太方便了——恰好就是这本书在写的这一年。I've seen this pattern before. 你把你想讨论的概念放在 timeline 的最近端，然后往前追溯两个阶段，给人一种 "这是历史的必然终点" 的感觉。

实际上：Claude Code 是 2025 年 2 月发布的。到我 "调研截止" 的 2026 年 4 月，也就 14 个月。用 14 个月定义一个工程时代——这个时间尺度我不舒服。Software 2.0 是我 2017 年写的，到 2024 年人们还在消化它的内涵。一个真正的工程范式转变需要更长的观察期。

**我的评价**：三阶段模型是一个优秀的教学工具，但不是历史描述。作为教学框架，它帮助读者理解 AI 编程的演进逻辑——这个价值是真实的。但作为历史分期，它过度简化了，而且 timeline 的端点恰好落在这本书的主题上——这个巧合让人怀疑。

文章的 5.2 节（能力叠加）试图弥补这个问题，值得肯定。但主图（timeline 286-302）还是暗示了一个干净的、线性的演进。**Timeline 图 vs 能力叠加的文字：这两者在打架。**

## 2. 四个支柱 vs 四个权衡：映射太干净了

这可能是整篇文章工程上最让我怀疑的地方。

```
"安全（Security）回应'效率 vs 安全'"
"可观测（Observability）回应'能力 vs 信任'"
"编排（Orchestration）回应'规模 vs 管理'"
"成本（Cost）回应'速度 vs 成本'"
```

这四个映射是一一对应的。每个 pillar 恰好对应一个 tension。一个 pillar 解决一个 tension。

**这个在真实工程里是不成立的。**

真实情况是：

- **安全（Security）**——它不只是回应 "效率 vs 安全"。它也深度影响可观测性（审计日志的粒度）、编排（安全策略决定了 Agent 可以或不可以并行做什么），甚至影响成本（安全检查本身消耗 Token 和计算）。
- **可观测（Observability）**——它不只是回应 "能力 vs 信任"。如果没有可观测性，你做不好编排调试（你不知道哪个 Agent 拖慢了流程），也算不清楚成本（你不知道 Token 花在了哪里）。
- **编排（Orchestration）**——它不只是回应 "规模 vs 管理"。编排直接影响成本（任务分解得越细，通信开销越大），也影响安全（Agent 之间的信任边界怎么定义）。
- **成本（Cost）**——它不只是回应 "速度 vs 成本"。成本决策做不好，安全控制可能被跳过（太贵了不做沙箱），可观测性可能被缩减（反正太贵了少记些日志）。

**每个 pillar 和每个 tension 之间都是多对多的关系。** 把复杂工程问题归纳成四个 1:1 的映射，这是一个 clean abstraction，符合教书的需求。但一个 cleaner abstraction 离 reality 更远。这是 engineering 里的 general tradeoff。

说实话，如果我是编辑，我会建议保留这个框架（它有助于记忆和理解），但加一个 "实际上这些映射不是一对一的" 的注释。

## 3. 四个支柱的顺序和优先级

"四个支柱缺一不可" — 文章在第 11 行和第 511 行两次强调这个说法。

让我用 march of nines 思维想一下：如果你只有 3 个支柱，系统会怎样？

- **没有编排**：单 Agent 模式，复杂任务做不了。你会卡在 Stage 2。
- **没有安全**：Agent 可能执行危险操作，需要人工监督兜底。
- **没有可观测**：出问题了不知道怎么回事，debug 靠猜。
- **没有成本**：Token 可能超支，但大不了多花点钱。

这四个的 impact 不是均衡的。在实践中：

| 优先级 | 支柱 | 理由 |
|--------|------|------|
| P0 | 编排 | 没有它你根本就做不了多 Agent 系统 |
| P1 | 安全 | 没有它你敢让 Agent 自主执行吗？ |
| P2 | 可观测 | 没有它你会在 debug 上浪费大量时间 |
| P3 | 成本 | 很重要，但短期内不会 kill 你的系统 |

**"缺一不可"——在数学意义上可能对。但在工程意义上，它们在不同阶段有完全不同的优先级。** 一家 startup 刚上 Agent 系统时不会去搭完整的成本归因体系；一个个人开发者可能三年不需要沙箱隔离。把四个支柱说成 "同等重要" 不是错误，但没有反映工程现实。

Martin Fowler 的五分类法也是类似——它对教学非常有用，但你要小心在工程讨论中用它做硬分类。文章自己也说了 "不同类别的工具解决不同类别的问题"、"可以组合使用"——这些补充很好。但五个类的定义框在工程边界上是模糊的。比如，Claude Code 现在既是 Terminal Agent 又有自己的编排（Project Instructions、CLAUDE.md），它到底是 Stage 3 还是 Stage 2？这种分类上的模糊性在工程实际中很常见。

## 4. 可观测性——最被低估的困难

> "可观测性通过运行追踪、状态监控、问题诊断三个层次，让 AI 编程从 '盲盒' 变成 '透明盒'。"

这个我直接质疑。

**AI 可观测性不是一个 "已有解决方案" 的问题。它是一个 open problem。**

LLM 调用链的 tracing——step 1 的输出是 step 2 的输入，Agent 的决策依赖关系是图状的而非链状的——现在的 observability 工具有几个能 handle 这个？Prometheus + Grafana 可以监控 API latency，但你能追踪到一个 Agent 为什么会选择调用 tool X 而不是 tool Y 吗？不能。

更关键的：LLM 的 "中间思考过程"（chain-of-thought, reasoning tokens）在当前的 observability 框架下几乎不可观测。你不知道模型的内部决策路径是什么。你说 "透明盒"——但 LLM 本身就是黑色的。我们最多能做到 "黑盒的输入输出被记录"，距离真正的 "透明盒" 还远。

这篇文章对可观测性的处理——我的感觉是乐观了 2-3 年。它描述的 "三个层次" 是目标，不是当前可用状态。

## 6. 具体建议（engineering realism 角度）

### 建议一：三阶段模型加一个 "说明" 框

在 timeline 图之后加一段话，直接说：
- 这些阶段的边界是模糊的
- Prompt engineering 没有在 2023 年结束
- "Harness Engineering" 这个命名是我的观点，不是一个工程史共识
- 我选择这个时间线是因为它有利于讲述这本书的叙事，但你要知道这是我的一己之见

这样反而更可信。诚实不是弱点。

### 建议二：四个支柱的映射关系应该承认多对多

当前 4×4 的映射表（principle-pillar mapping）太干净了。建议改为：
- 主映射保留（p incipal mapping to primary pillar）
- 加一列 "也影响"（also affects），列出每个 pillar 还影响哪些 tension

### 建议三：可观测性部分应该区分 "目标" 和 "现状"

当前的内容描述的是可观测性的理想状态。建议明确分两段：
1. **目标**（the north star）：运行追踪、状态监控、问题诊断
2. **现状**（current reality）：LLM 的内部推理仍不可观测，Agent 决策路径追踪还在早期，多数工具只有基础日志

### 建议四：加上 "什么时候不需要 Harness Engineering"

整篇文章一直在说 "你需要这个"。但如果我是一个写 500 行脚本的 solo developer，我真的需要四个支柱吗？答案是 no。

明确写出这个框架的适用边界——比如 "如果你的 AI 编程场景满足以下条件，你可能只需要 Stage 1 或 Stage 2 的能力：单文件任务、个人项目、容忍出错"。这会增加可信度。

## 总结

这是一篇诚实的理论框架文章。它写得比大多数同类文章好——因为它确实在试图解释真实工程问题，而不是在画大饼。四个支柱的选题是正确的，Martin Fowler 的分类法是实用的，AI 编程的核心张力是真实的。

但它有一个根本性的 tension（pun intended）：**作为一本书的理论框架，它必须把自己包装得漂亮、完整、自洽。而作为工程 reality，真实世界从来都不漂亮、不完整、不自洽。**

具体来说：
- 三阶段演进模型的 timeline——文章的叙事利益和工程 reality 之间有一个 gap
- 四个支柱与四个张力的一一映射——太干净了，不真实
- 可观测性的当前状态——被显著高估
- "缺一不可" 的说法——忽略了实际工程部署中的优先级差异

这些都不是致命问题。对于一本面向开发者入门到中级读者的书来说，这个框架已经足够好。但如果你问我 "作为 Karpathy，你会给你团队的 junior engineer 读这个框架来理解 AI 编程工程化吗？"——我会说，读，但要带着 grain of salt。

就这样。
