# HEDQ 质量评价标准

> **Harness Engineering Documentation Quality** — 8 维书籍内容质量评价体系
> **版本**: v2.0 | **日期**: 2026-06-29
> **设计依据**: Darwin SKILL 9 维 Rubric (SkillLens arXiv 2605.23899) + ELISA 文档评估 + CNCF TechDocs + AGENTS.md 规范
> **核心对齐**: AGENTS.md（品牌名、链接、代码块、文风、Mermaid 配色）

---

## 一、体系概览

HEDQ 将书籍内容质量分解为 **8 个维度**，总分 **100 分**，三个评估层面：

```
内容层 (45分)      呈现层 (35分)      实用层 (20分)
┌─────────────┐  ┌─────────────┐   ┌─────────────┐
│ D2 准确性 20 │  │ D1 结构  15 │   │ D4 实操  15 │
│ D3 角色   15 │  │ D6 文风  10 │   │ D8 图表   5 │
│ D5 反面   10 │  │ D7 术语  10 │   └─────────────┘
└─────────────┘  └─────────────┘
```

| 维度 | 名称 | 权重 | 自动检测满分 | 评估层 | 自动化率 |
|------|------|:----:|:----------:|:------:|:--------:|
| D1 | 结构与元数据规范性 | 15 | **14**¹ | 呈现层 | ~95% |
| D2 | 内容准确性与时效性 | 20 | **6**² | 内容层 | ~50% |
| D3 | 读者角色覆盖与导航 | 15 | **6**³ | 内容层 | ~40% |
| D4 | 代码块格式与注释覆盖率 | 15 | **4**⁴ | 实用层 | ~60% |
| D5 | 反面案例与边界条件 | 10 | **13**⁵ | 内容层 | ~65% |
| D6 | 文风与可读性 | 10 | **2**⁶ | 呈现层 | ~30% |
| D7 | 术语与品牌一致性 | 10 | 10 | 呈现层 | ~95% |
| D8 | 图表与可视化质量 | 5 | **3.5**⁷ | 实用层 | ~40% |
| **合计** | | **100** | **58.5¹** | | **~60%** |

> **注**：自动检测满分 ≠ 标准定义权重。`run-hedq.py` 输出层已统一为 100 分制，此处 58.5 为内部检测原始满分。
> ¹ 各维度内部原始满分：D1(14)+D2(6)+D3(6)+D4(4)+D5(13)+D6(2)+D7(10)+D8(3.5)=58.5，输出时按系数 `k=100/58.5` 缩放。部分维度的自动化检测仅覆盖其子项的子集：
> ¹ D1 自动化检测覆盖 D1.1（SUMMARY 路径）、D1.2（内部链接）、D1.4（品牌名）、D1.5（链接文字与 H1 一致性），未覆盖 D1.3（代码块 `:path` 注释，该检查归入 D4.1）。
> ² D2 自动化检测仅覆盖 D2.2（版本新鲜度），未覆盖 D2.1（技术事实验证，需 deep-research）和 D2.3（源材料映射）。
> ³ D3 自动化检测覆盖 D3.1（追溯矩阵覆盖率），未覆盖 D3.2（阅读路径）和 D3.3（信息分类）。
> ⁴ D4 自动化检测覆盖 D4.1（三层检查体系：缺:path/路径无效/自引用/白名单warn），未覆盖 D4.2（配置文件）、D4.3（步骤重现）、D4.4（代码可运行）。
> ⁵ D5 新增 D5.4（内容厚度检查），覆盖反模式/失败场景/边界条件章节下的段落数验证。
> ⁶ D6 自动化检测覆盖 D6.3（AI 腔检测），未覆盖 D6.1（说人话）、D6.2（段落收益）、D6.4（简洁性）。
> ⁷ D8 自动化检测覆盖 D8.1（Mermaid 语法）和 D8.2（配色合规）。D8.3（图表理解辅助性）未自动化，由 d8_render.py 提供渲染验证代理检测（max=0，不评分）。

## 二、评分等级

| 等级 | 分数 | 含义 |
|:----:|:----:|------|
| 🟢 **READY** | ≥90 | 可发布，无需修改 |
| 🟡 **CONDITIONAL** | 75–89 | 有条件发布，修复 P1 问题后即可 |
| 🟠 **NEEDS WORK** | 60–74 | 需修改，有 P0/P1 问题 |
| 🔴 **DRAFT** | <60 | 不可发布，需大幅重写 |

### P0 一票降级规则

若任何维度存在 **P0 级违规**（关键事实错误、不存在的 API、无效配置、断裂的核心链接），
则最终评级**强制降一级**：

| 原始评级 | 降级后评级 |
|:--------:|:----------:|
| READY | CONDITIONAL |
| CONDITIONAL | NEEDS WORK |
| NEEDS WORK | DRAFT |
| DRAFT | DRAFT |

此规则确保 P0 问题不会被高分维度掩盖。

## 三、D1 — 结构与元数据规范性（15 分）

### 子项与权重

| 子项 ID | 子项名称 | 满分 | 检查方法 |
|---------|---------|:----:|---------|
| D1.1 | SUMMARY.md 路径完整性 | 3 | 遍历 SUMMARY.md 中所有 `(路径.md)`，检查磁盘文件存在 |
| D1.2 | 正文内部链接有效性 | 4 | 提取全文 `.md` 链接，解析相对路径，检查目标存在 |
| D1.3 | 代码块 `language:path` 格式 | 4 | 非 Mermaid 代码块必须有 `language:相对路径` 格式（此子项归入 D4.1） |
| D1.4 | 品牌名与规范拼写 | 4 | 检查 OpenCode / oh-my-openagent / MCP / mdBook |
| D1.5 | 交叉引用标题一致性 | 3 | 链接文字与目标文件 H1 标题一致（自动化：`checks/d1_structural.py`） |

### 扣分规则

| 违规 | 扣分 |
|------|:----:|
| SUMMARY.md 中路径目标不存在 | 每条扣 1 分 |
| 正文内部链接断链 | 每条扣 1 分（同一文件重复不计） |
| 非 Mermaid 代码块缺 `:path` | 每个扣 0.5 分（归入 D4.1） |
| 品牌名拼写错误 | 每处扣 0.5 分 |
| 链接文字与目标 H1 不匹配 | 按比例扣分（D1.5 = 3 × 匹配比例） |

### 计算方式

```
D1 = D1.1(0-3) + D1.2(0-4) + D1.4(0-4) + D1.5(0-3)
```

### 打分示例

```text
D1.1: SUMMARY.md 67/67 路径存在  → 3 × 1.0 = 3.0
D1.2: 1092/1094 链接有效 (2 个断链) → 4 × 0.998 = 3.99 ≈ 3.0
D1.4: 1 处 "Opencode" → 扣 0.5 → 4.0 - 0.5 = 3.5
D1.5: 317 处交叉引用, 12 处链接文字与 H1 不匹配 → 3 × (305/317) = 2.9

D1 = 3.0 + 3.0 + 3.5 + 2.9 = 12.4 / 15
```

## 五、D3 — 读者角色覆盖与导航（15 分）

### 子项与权重

| 子项 ID | 子项名称 | 满分 | 检查方法 |
|---------|---------|:----:|---------|
| D3.1 | 用户故事实质覆盖 | 6 | 逐条评估 55 个故事覆盖深度 |
| D3.2 | 阅读路径可操作性 | 5 | 5 个高优角色路径实操验证 |
| D3.3 | 信息类型分类 | 4 | 每篇明确标注概念/操作/参考类型 |

### 覆盖深度评分标准

| 评级 | 定义 | 分值 |
|:----:|------|:----:|
| COMPLETE | 用户故事被实质满足 | 5 |
| SUBSTANTIAL | 映射存在但深度不足 | 3 |
| PARTIAL | 有相关内容但缺口明显 | 1 |
| MISSING | 无任何内容覆盖 | 0 |

### 计算方式

```
故事覆盖率 = Σ(各故事深度分) / (55 × 5)
D3.1 = 6 × 故事覆盖率
D3.2 = 5 × (可操作路径数 / 5)
D3.3 = 4 × (有类型标注篇数 / 总篇数)
D3 = D3.1 + D3.2 + D3.3
```

### 打分示例

```text
D3.1: 14 角色 × 55 故事, 加权覆盖率 = 80%
      D3.1 = 6 × 0.80 = 4.8
D3.2: 5 条阅读路径, 4 条可操作 → 5 × 0.8 = 4.0
D3.3: 56 篇中 48 篇有类型标注 → 4 × 0.857 = 3.4

D3 = 4.8 + 4.0 + 3.4 = 12.2 / 15
```

## 六、D4 — 代码块格式与注释覆盖率（15 分）

### 子项与权重

| 子项 ID | 子项名称 | 满分 | 检查方法 |
|---------|---------|:----:|---------|
| D4.1 | 代码块 `language:path` 注释质量 | 4 | 三层检查体系（自动化：`checks/d4_codeblocks.py` v2.0） |
| D4.2 | 配置文件直接可用 | 5 | examples/ 文件有版本标注 + 说明文档（**未自动化**） |
| D4.3 | 步骤可重现 | 4 | 按步骤式文章操作到预期结果（**未自动化**） |
| D4.4 | 代码示例可运行 | 2 | 提取代码块，逐块执行（子 agent / bash，**未自动化**） |

> **注**：D4 原名"实操可执行性"，但自动化检测仅覆盖 D4.1（代码块格式注释），
> 完整的代码可运行性检测需人工介入。故维度名称调整为"代码块格式与注释覆盖率"以准确反映自动化范围。
> D4.2/D4.3/D4.4 的权重合计 11 分未纳入自动检测，仅 D4.1 的 4 分计入自动检测口径。

### D4.1 三层检查体系（v2.0）

```
Layer 1 — 缺 :path 检测：非白名单块缺 :path 注释 → -0.3/个（原核心检查）
Layer 2 — 自引用检测：仅报告，不计扣分（技术书籍中自引用是合理的设计约束）
Layer 3 — 白名单约定路径报告：统计 :terminal 等约定路径，仅报告不计扣分
```

设计约定路径（DESIGN_CONVENTIONS）：`terminal`、`terminal-session.md` 等属于全书统一的视觉约定，不计入扣分但会在报告中统计。
全书白名单块携带的示意路径（如 `json:opencode.json`）同样仅统计不计扣分，它们属于惯用的代码块标识方式。

### 代码块格式评分

```text
D4.1 = max(0, 4.0 - 缺:path数 × 0.3)
```

白名单类型（无需 path 注释）：mermaid、bash、sh、shell、console、text、json、csv、toml、yaml、yml、markdown、dockerfile、html、css。

### 打分示例

```text
D4.1: 1453 个代码块
        - 1 个非白名单块缺 :path → -1 × 0.3 = -0.3
        - 93 个自引用（仅报告，不计扣分）
        - 428 个约定路径 :terminal（仅报告，不计扣分）
        - 527 个白名单示意路径（仅报告，不计扣分）
        D4.1 = max(0, 4 - 0.3) = 3.7

D4.2: examples/ 36 个文件, 25 个有版本标注（人工检查）
      5 × (25/36) = 3.5
D4.3: 5 篇步骤式文章, 4 篇可完整重现（人工检查）
      4 × 0.8 = 3.2
D4.4: 12 个代码块, 10 个可运行, 2 个缺依赖（人工检查）
      2 × (10/12) = 1.7

D4 = 1.8 + 3.5 + 3.2 + 1.7 = 10.2 / 15
（自动检测口径仅 D4.1 = 1.8 / 4）
```

## 七、D5 — 反面案例与边界条件（10 分）

### 子项与权重

| 子项 ID | 子项名称 | 满分 | 检查方法 |
|---------|---------|:----:|---------|
| D5.1 | 反模式清单 | 4 | 实操类文章是否有"不要这样做"章节 |
| D5.2 | 失败场景分析 | 3 | 是否包含"什么情况下会失败"、"常见错误" |
| D5.3 | 边界条件讨论 | 3 | 是否提及限制条件、"不适用场景" |
| D5.4 | 反面案例内容厚度 | 3 | 已识别到的反模式/失败/边界章节是否包含 ≥3 段实质内容（自动化：`checks/d5_antipatterns.py`） |

### 判断阈值

```text
覆盖率 = 覆盖全部 3 项的文章数 / 实操类文章总数
≥80% → 满分
≥60% → 80% 分
<60% → 50% 分

内容厚度 = 1 - (浅层章节数 / 总章节数)，浅层 = 段落 < 3 段
D5.4 = 3 × 内容厚度
```

### 打分示例

```text
20 篇实操类文章:
  16 篇有反模式章节 (80%)  → D5.1 = 4.0
  14 篇有失败场景 (70%)    → D5.2 = 3 × 0.70 = 2.1
  12 篇有边界条件 (60%)    → D5.3 = 3 × 0.60 = 1.8
  42 个章节, 8 个浅层（<3段）→ D5.4 = 3 × (1-8/42) = 2.4

D5 = 4.0 + 2.1 + 1.8 + 2.4 = 10.3 / 13
```

## 八、D6 — 文风与可读性（10 分）

### 子项与权重

| 子项 ID | 子项名称 | 满分 | 检查方法 |
|---------|---------|:----:|---------|
| D6.1 | "说人话"原则 | 3 | 优先具体操作语言，避免抽象隐喻 |
| D6.2 | 每段回答读者问题 | 3 | 每段话有明确读者收益 |
| D6.3 | 避免 AI 腔废话 | 2 | 禁用词库扫描（全自动） |
| D6.4 | 简洁性 | 2 | 段落平均长度 ≤ 5 句 |

### 禁用词库（自动检查）

禁用词清单统一维护在独立文件中：
`scripts/qa/hedq/config/banned_ai_speak.txt`（42 条模式，按五类组织）

| 类别 | 包含模式 | 条数 |
|------|----------|:----:|
| P0 废话开场/空枢纽句 | 说白了、换句话说、值得注意的是、不言而喻、显而易见、众所周知、毫无疑问、需要注意的是、值得一提的是、不可否认、不可置否、不得不承认、不容忽视、不争的事实、毋庸置疑 | 15 |
| P1 总结套话 | 综上所述、综上、基于此、因此可以说、简单来说、换言之 | 6 |
| P1 逻辑过渡/强调 | 更为重要的是、需要指出的是、需要强调的是、从本质上讲、从另一个角度、不难发现、让我们来看看 | 7 |
| P2 模糊限定/过度绝对 | 在一定程度上、从某种角度来说、在某种程度上、从某种意义上说、极其重要、不可或缺、具有重要意义、扮演着重要角色、发挥着重要的作用、起着至关重要的作用、具有深远的意义 | 11 |
| P2 元评论/AI尾气 | 正如前文所述、如上所述、引起了广泛关注 | 3 |

> **注**：`显而易见` 使用负向回视断言 `(?<!非)显而易见`，排除合法技术用语"非显而易见"。
> 正常过渡词（首先、其次、再次）不在检查列表中，其属于中文写作的正常组成部分。
> 变长通配模式（在...方面、在...层面）也未纳入，因 Python re 不支持变长回视。
> 如需新增/移除禁用词，直接编辑 `config/banned_ai_speak.txt` 即可，无需修改检测代码。

### 扣分规则

```
D6.3: 命中每个禁用词扣 0.2 分, 扣完为止
```

### 打分示例

```text
D6.1: 17 段中 14 段有明确操作语言 → 3 × (14/17) = 2.5
D6.2: 17 段中 14 段有明确读者收益 → 3 × (14/17) = 2.5
D6.3: 0 处命中禁用词 → 2.0
D6.4: 平均 4.2 句/段, 无重复段 → 2.0

D6 = 2.5 + 2.5 + 2.0 + 2.0 = 9.0 / 10
```

## 十、D8 — 图表与可视化质量（5 分）

### 子项与权重

| 子项 ID | 子项名称 | 满分 | 检查方法 |
|---------|---------|:----:|---------|
| D8.1 | Mermaid 语法正确 | 2 | 检查波浪号引用、方向声明（自动化：`checks/d8_mermaid.py`） |

### 评估可信度与子 agent 验证

评分结果的可信度取决于谁在执行评分。为消除同上下文自评的乐观偏误（SkillLens 实证 LLM-as-judge 自评准确率仅 46.4%）：

| 评分场景 | 执行者 | 可信度 | 说明 |
|:--------:|:------:|:------:|------|
| **自动化 CLI** | `run-hedq.py` | 高 | Python 脚本确定性输出，无偏误。此为所有评分的基准。 |
| **主 agent 解读报告** | 主 AI agent | 中 | 解读报告本身无偏误，但自评"是否修复成功"时有乐观偏误风险。 |
| **主 agent 自评修复效果** | 主 AI agent | 低（46.4%） | 同上下文自评修复前后分数，LLM 倾向于高估改进幅度。**禁止**。 |
| **独立子 agent 评分** | Spawn 的子 agent | 中高 | 独立上下文无偏误诱导，但受子 agent 自身能力影响。**强制使用**。 |
| **人工裁断** | 人类 | 最高 | 当子 agent 与主 agent 评分差异 >15% 时，最终裁断权归人类。 |

**规则**：所有 Verify 阶段的评分必须由独立子 agent 执行。若子 agent 不可用，需在报告中标注 `dry_run`，分数加 ⚠️ 标记。dry_run 比例 >30% 时评估结果整体不可信。
